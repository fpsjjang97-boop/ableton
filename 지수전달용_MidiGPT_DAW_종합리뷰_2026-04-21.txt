지수 전달용: MidiGPT + DAW 종합 리뷰
작성일: 2026-04-21

0. 이 문서의 목적

이 문서는 지금까지 따로 정리되어 있던 리뷰들을 하나로 합친 종합판입니다.
목적은 아래 4가지입니다.

1) 사용자가 실제로 원하는 제품이 무엇인지 정확히 다시 정의
2) 현재 MidiGPT/DAW 구조가 그 목표와 어디서 어긋나는지 정리
3) 최근 업데이트에서 해결된 것과 아직 남은 문제를 분리
4) 적은 데이터 조건에서도 현실적으로 제품을 앞으로 밀 수 있는 구조와 계획 제안

이 문서는 "단순 모델 품질 평가" 문서가 아니라, "제품 목표 대비 현재 구조 리뷰 + 다음 설계 제안" 문서로 보는 것이 맞습니다.


1. 한 줄 결론

현재 모델은 여전히 "주어진 심볼 시퀀스를 이어서 만드는 symbolic continuation/variation 모델" 성격이 강합니다.
반면 사용자가 원하는 최종 제품은 "DAW 안에서 특정 구간, 특정 트랙을 문맥을 보고 채워주는 작곡가용 커서"입니다.

즉 지금 가장 큰 문제는 단순히 학습량 부족만이 아니라,

- 제품 목표와 모델 학습 목표가 아직 완전히 맞지 않고
- DAW UI가 보내는 정보와 서버/모델이 실제로 쓰는 정보가 다르고
- 트랙/역할/보이싱/구간 편집 개념이 모델 내부 표현으로 충분히 올라오지 못하고 있다는 점입니다.

최근 업데이트는 분명 의미가 있었습니다.
특히 chord conditioning 보강, continuation SFT 분포 수정, unified composer 방향성 추가는 모두 맞는 수정입니다.
다만 아직은 "완전한 해결"이 아니라 "올바른 방향으로 일부 전진"에 가깝습니다.


2. 사용자가 실제로 원하는 제품 정의

사용자가 원하는 최종 제품은 Suno류의 "통 wav/mp3 생성기"가 아닙니다.
원하는 것은 아래와 같은 DAW 동반형 작곡 도구입니다.

2-1. 단기 목표

- 사용자가 DAW에서 이미 작업 중인 곡이 있음
- 멜로디, 피아노, 코드, 베이스, 드럼, 스트링, 기타 등 일부 트랙은 이미 있고 일부는 비어 있거나 부족함
- 사용자가 원하는 구간을 드래그하고, 원하는 타겟 트랙을 지정함
- AI에게 "이 구간의 드럼/베이스/피아노/스트링/기타를 만들어줘" 혹은 "이 구간만 더 자연스럽게 바꿔줘"라고 요청
- AI는 기존 트랙들, 섹션, 장르, BPM, 코드 흐름, 주변 문맥을 참고해 선택된 구간의 선택된 파트를 MIDI 노트로 생성/편집
- 피아노/멜로디를 보고 bass, drums, strings, guitar 등 다른 파트를 채워줄 수 있어야 함
- 특정 구간만 re-harmonization, fill-in, variation, density control이 가능해야 함
- 단순 continuation이 아니라 in-place editing / infill / track-targeted generation을 지원해야 함
- 결과는 DAW에서 바로 수정 가능한 형태로 들어옴

즉 단기 목표의 본질은 특정 악기 하나가 아니라,
"선택 구간 + 선택 타겟 트랙 + 기존 문맥 기반 편집/보조 생성"을 모든 주요 악기군에 대해 가능하게 만드는 것입니다.
드럼 생성은 이 비전을 설명하는 예시 중 하나일 뿐, 단기 목표 전체를 의미하지는 않습니다.

2-2. 장기 목표

- 사용자의 실제 작업 데이터를 DAW에서 캡처
- 그 데이터를 통해 점점 사용자의 취향과 습관을 반영
- 결국 "내 복제본 작곡가", "옆에 앉은 작은 작곡가", "작곡가용 Cursor" 같은 경험 제공
- MIDI뿐 아니라 샘플/wav 사용 맥락도 이해하도록 확장


3. 현재 구현된 것 요약

지금 시스템에는 이미 좋은 재료가 꽤 많습니다.

3-1. 이미 있는 것

- 자체 symbolic music model (MidiGPT)
- DAW 프로젝트/플러그인 측 UI
- inference server
- LoRA 학습/교체 경로
- reviewer 관련 코드
- unified composer 방향 코드
- rule-based composer 자산
- audio2midi / sample 관련 확장 방향 코드

3-2. 즉, 완전히 없는 상태는 아님

현재는 "재료는 꽤 모였는데, 제품 목표 기준으로 하나의 폐쇄 루프(closed loop)로 아직 연결되지 않은 상태"에 가깝습니다.


4. 최근 업데이트에서 실제로 좋아진 것

최근 업데이트를 기준으로 확인된 개선점은 아래와 같습니다.

4-1. chord conditioning은 일부 좋아짐

예전보다 초반 몇 마디에서 "이 마디가 C인지 Am인지" 같은 코드감이 더 들리는 경우가 있었습니다.
즉 코드톤을 더 강하게 유도하려는 수정은 실제로 일정 부분 효과가 있었습니다.

4-2. continuation SFT 분포 수정은 맞는 수정

기존 continuation SFT는 split 이후 output이 time 0부터 시작하도록 rebase되는 문제가 있었고,
이로 인해 학습 시 SEP 다음에 항상 Bar_0 근처 패턴을 보게 되는 분포 왜곡이 있었습니다.
이 부분을 수정한 방향은 타당합니다.

4-3. unified composer / reviewer 방향성은 맞음

아직 완성도는 낮지만,
"그냥 next-token 생성 하나로 다 해결하려 하지 말고 planner/reviewer/hybrid composer 쪽으로 가자"
라는 방향 자체는 사용자가 원하는 제품 목표와 더 가깝습니다.


5. 최근 업데이트 후에도 남아 있는 핵심 문제

업데이트가 의미 있었던 것은 맞지만, 아래 문제들은 아직 해결되지 않았습니다.

5-1. 코드 진행이 항상 안정적으로 들리지는 않음

초반 마디는 나아진 경우가 있지만, 후반으로 갈수록 코드 중심이 다시 흐려지는 구간이 여전히 있습니다.

5-2. 피아노 보이싱이 여전히 사람답지 않음

한 트랙 안에 너무 많은 음이 동시에 쌓이거나,
저음/중음/고음 역할이 분리되지 않은 채 뭉치면서 사람이 누를 수 있는 피아노 반주처럼 들리지 않는 경우가 남아 있습니다.

5-3. 한 트랙 안에 여러 역할이 계속 섞임

accomp 트랙이 피아노 comp 역할만 하는 것이 아니라,
저음 받침, pad성 화성, string성 sustain, 장식음 비슷한 요소까지 같이 들어가는 경우가 있습니다.

5-4. 생성 길이는 길어 보여도 실제 중간 마디가 비는 현상 남음

파일 길이는 64마디처럼 나와도 실제 음은 첫 부분과 끝 부분에만 몰리고 중간은 사실상 비어 있는 결과가 확인되었습니다.


6. 빈 마디 문제: 무엇이 맞았고, 무엇이 아직 안 끝났는가

빈 마디 문제는 한 가지 원인만 있었던 것이 아니라 최소 두 겹으로 보는 것이 맞습니다.

6-1. 지수가 짚었던 원인은 맞았음

기존 continuation SFT에서 split 이후 output time rebase가 들어가면,
학습 시 모델은 SEP 뒤에 항상 Bar_0 근처를 보게 됩니다.
그런데 실제 추론에서는 input 마지막 bar가 Bar_5, Bar_8, Bar_12일 수 있습니다.

이때 모델은 학습 습관대로 Bar_0 쪽을 내보내려 하고,
FSM/constraint는 backward jump를 막고,
결과적으로 모델이 꼬여서 공백/점프/이상 출력이 생길 수 있습니다.

즉 "학습/추론 bar 분포 불일치"를 원인으로 본 진단은 맞았습니다.

관련 코드:
- midigpt/build_sft_pairs.py

6-2. 그런데 그 수정만으로 공백이 완치되지는 않았음

최신 코드로 다시 학습한 뒤에도,
"길이는 길게 나오는데 실제 음은 일부 마디에만 있는 결과"가 여전히 나왔습니다.

이것은 단순히 SEP 뒤 bar numbering 문제만이 아니라,
현재 추론 제약이 아래 정도만 강하게 관리하기 때문입니다.

- bar가 뒤로 가지 말 것
- 너무 큰 jump를 하지 말 것
- 너무 빨리 EOS로 끝내지 말 것

반면 아래는 아직 약합니다.

- 각 마디를 실제로 음으로 채워야 한다
- 긴 연속 empty bar를 만들면 안 된다
- target range 안에서는 최소 밀도를 유지해야 한다

즉 현재 생성기는 "bar 번호 질서"는 어느 정도 관리하지만,
"마디 내부가 실제로 비어 있지 않도록" 강하게 보장하지는 못합니다.

관련 코드:
- midigpt/inference/constrained.py
- midigpt/inference/engine.py

6-3. 결론

정리하면:

- 지수가 수정한 원인은 실제 원인 중 하나였고, 그 수정은 의미가 있다.
- 하지만 지금 남은 공백 문제는 "마디 번호 분포 문제"만이 아니라
  "빈 마디를 허용하는 생성 규칙과 데이터/목표 설계 문제"까지 함께 있다.


7. 트랙명/악기 인식 문제: 무엇이 진짜 문제인가

사용자는 violin1, violin_legato 같은 이름이면 당연히 strings로 인식되어야 한다고 봅니다.
실제 코드 기준으로 입력 단계에서는 이 부분이 맞게 동작합니다.

7-1. 입력 단계에서 violin 계열은 실제로 strings로 분류됨

확인 결과:

- VIOLIN_LEGATO
- VIOLIN2_LEGATO

같은 트랙명은 입력 시 strings 카테고리로 잘 들어갑니다.

관련 코드:
- midigpt/tokenizer/encoder.py

7-2. 그런데 왜 결과에서는 strings가 accomp로 섞여 보이는가

문제는 "입력 읽기 실패"가 아니라 "생성 시 트랙 identity 유지 실패" 쪽입니다.

현재 토크나이저는 원래 트랙명을 계속 보존하지 않고,
결국 violin1 같은 이름도 strings라는 큰 카테고리로 접습니다.

즉 모델은 "원래 이 트랙 이름이 violin1이었다"는 세밀한 identity를 계속 들고 가지 않습니다.

7-3. decoder 기본 트랙이 accomp라는 점도 문제를 키움

현재 decoder는 트랙 토큰이 명시되지 않은 상태에서 기본 current_track을 accomp로 시작합니다.
그래서 모델이 Track_strings를 분명히 내주지 못하면,
원래 strings 성격의 음도 accomp로 들어갈 수 있습니다.

관련 코드:
- midigpt/tokenizer/decoder.py

7-4. 결론

지금 문제는 "violin 이름을 못 읽는다"가 아니라:

- 입력에서는 strings로 읽지만
- 원래 상세 identity는 유지되지 않고
- 생성에서 Track_strings 선언이 약하면
- decoder 기본값인 accomp로 떨어져 섞일 수 있다는 구조 문제입니다.


8. 현재 구조가 제품 목표와 어긋나는 핵심 이유

여기가 가장 중요합니다.
사용자가 원하는 제품은 "구간 선택형, 타겟 트랙형, 문맥 조건형 편집 모델"인데,
현재 시스템은 아직 그 구조로 완전히 서 있지 않습니다.

8-1. 생성 단위가 아직 too big sequence 생성에 가깝다

현재 모델은 본질적으로 next-token 기반 continuation/variation 성격이 강합니다.
이 구조는 "뒤를 이어 쓰기"에는 자연스럽지만,
"8~12마디의 drums만 채우기", "selected bars의 strings만 보강" 같은 편집 작업에는 직접적으로 맞지 않습니다.

8-2. conditioning 정보가 너무 약하다

현재 SongMeta 계층은 대체로 key/style/section/tempo 수준에 가깝습니다.
사용자가 실제로 원하는 생성은 아래 정보를 더 강하게 요구합니다.

- target track
- target bar range
- 앞/뒤 섹션 문맥
- 다른 트랙의 역할
- chord map
- groove / density / energy
- register budget
- melodic anchor

즉 현재는 "곡 맥락"보다 "짧은 메타데이터"에 더 가깝습니다.

8-3. 역할(role) 정보가 명시적이지 않다

사용자는 단순히 "이건 피아노 트랙이다"보다 더 많은 것을 원합니다.

- melody인가
- counter melody인가
- accompaniment인가
- bass foundation인가
- rhythmic hook인가
- pad sustain인가

현재 모델은 악기 카테고리 정도는 보지만,
"이 트랙이 곡 안에서 무슨 역할을 하는가"는 충분히 명시적으로 배우지 못합니다.

8-4. 피아노 보이싱 / 악기 물리성 제약이 약하다

사람이 칠 수 있는 보이싱, 손 분배, 음역 배치, 동시에 누를 수 있는 음 수, strong beat 안정성 같은 규칙이 약합니다.
그래서 "화음은 어느 정도 맞추려다가도 피아노답지 않은 뭉치"가 나옵니다.

8-5. 멀티트랙 생성 순서 전략이 약하다

실전 편곡은 보통:

- melody / chords
- piano or harmonic comp
- bass
- drums
- pad / strings / fills

같은 순서와 의존성을 갖습니다.
현재는 이런 shared context 기반 순차 생성 전략이 완전히 자리잡지 못했습니다.

8-6. DAW가 보내는 정보와 서버가 실제로 쓰는 정보가 다르다

이건 제품 관점에서 매우 큰 문제입니다.

현재 DAW 쪽 UI/AIBridge는 task, start_bar, end_bar 같은 정보를 보내는 방향이 이미 있습니다.
즉 프론트는 "bar infill", "track completion", "구간 지정" 같은 UX를 상정하고 있습니다.

그런데 inference server/request 계층은 아직 그 정보를 완전히 받아서 generation path에 반영하지 않습니다.
즉 UI는 편집 모델처럼 보이는데, 실제 서버는 아직 variation/continuation 쪽에 더 가깝게 동작합니다.

관련 코드:
- juce_daw_clean/Source/AI/AIPanel.cpp
- juce_daw_clean/Source/AI/AIBridge.cpp
- midigpt/inference_server.py

8-7. DAW 결과 반영 방식이 append 중심이다

사용자가 원하는 것은 "선택 구간 교체/삽입"인데,
현재 흐름은 생성 결과를 트랙 뒤쪽에 append하는 경향이 강합니다.

즉 "in-place edit"가 아니라 "뒤에 새 클립 추가"에 더 가깝습니다.
이 차이는 제품 경험에서 매우 큽니다.

8-8. unified composer는 방향은 맞지만 아직 scaffold에 가깝다

unified composer는 현재 "우리가 결국 이런 구조로 가야 한다"는 신호로는 매우 좋습니다.
하지만 지금 당장 완전한 planner/router/reviewer/generator 파이프라인으로 폐쇄 루프를 이루고 있다고 보기는 어렵습니다.

관련 코드:
- agents/unified_composer.py

8-9. reviewer는 있지만 generation loop를 완전히 닫지 못한다

reviewer 자체 아이디어는 맞습니다.
하지만 현재는

- 생성
- 리뷰
- 낮은 점수면 수정/재생성

같은 완전한 자동 루프가 충분히 안정적으로 연결되어 있다고 보기 어렵습니다.

관련 코드:
- agents/reviewer.py


9. 지금 구현된 것 vs 사용자가 원하는 것 비교

9-1. 지금 잘 되어 있는 것

- symbolic MIDI 기반 자체 모델이 있다
- DAW 안에서 AI 기능을 호출하는 UX 뼈대가 있다
- inference server가 있다
- LoRA/hotswap 방향이 있다
- rule-based generator 자산이 있다
- reviewer 개념이 있다
- audio2midi 확장 방향 코드가 있다

9-2. 지금 부족한 것

- selected range 편집을 진짜로 수행하는 generation path
- target track를 강하게 반영하는 conditioning
- melody/piano/chords/context를 보고 악기군별로 target track을 채우는 task-specific model behavior
- track role 명시 데이터
- song-level context builder
- human-playable voicing constraint
- empty-bar rejection / regeneration loop
- user personalization loop

9-3. 즉 현재 상태를 한 문장으로 말하면

"음악 관련 여러 기능을 가진 코드베이스"는 맞지만,
"사용자가 원하는 DAW 작곡 커서 제품"으로는 아직 핵심 루프가 덜 연결된 상태입니다.


10. 지금 당장 가장 현실적인 제품 목표

모든 것을 한 번에 완성도 높게 하려 하면 구조가 계속 흔들릴 가능성이 큽니다.
하지만 그렇다고 제품 목표 자체를 드럼 하나로 축소하면 프로젝트 의미가 줄어듭니다.
따라서 아래처럼 "제품 목표"와 "첫 검증 태스크"를 분리해서 보는 것이 맞습니다.

10-1. 제품 단기 목표

"사용자가 DAW에서 선택한 구간의 선택한 악기 파트를, 기존 문맥을 참고해서 생성/편집해주는 기능을 주요 악기군 전반에 대해 제공한다."

여기서 주요 악기군은 최소 아래를 포함하는 것이 맞습니다.

- drums
- bass
- piano / accompaniment
- strings / pad
- guitar / pluck 계열

즉 단기 목표는 이미 "다양한 트랙을 문맥 기반으로 채워주는 DAW 편집형 작곡 도구"여야 합니다.

10-2. 첫 검증 태스크(추천)

그 위 비전 안에서, 첫 번째로 품질/구조를 검증하기 쉬운 태스크를 하나 고르자면
"melody + piano context를 보고 selected bars의 drums를 생성"
이 여전히 좋은 첫 검증 예시가 될 수 있습니다.

중요한 점은 이것이 제품 범위 그 자체가 아니라,
"모든 악기군으로 확장될 편집형 구조를 처음 검증하는 대표 태스크"라는 점입니다.

10-3. 성공 기준

- 선택한 bar range 안에만 결과가 들어갈 것
- target track가 명확히 유지될 것
- 기존 다른 트랙들과 역할 충돌이 적을 것
- 섹션에 맞는 density를 유지할 것
- empty bar가 거의 없을 것
- 사용자가 바로 손으로 수정 가능할 것

이 성공 기준은 드럼뿐 아니라 bass/piano/strings/guitar에도 공통으로 적용되어야 합니다.


11. 적은 데이터 문제: 현실적으로 어떻게 풀 것인가

사용자 조건상 작곡가 2명이 직접 만드는 MIDI 데이터는 매우 귀합니다.
따라서 "몇천~몇만 개를 먼저 만들자"는 전략은 현실성이 낮습니다.

이 프로젝트는 처음부터 "저데이터 효율"을 전제로 설계하는 것이 맞습니다.

11-1. 중요한 전환

많은 generic MIDI를 무작정 모으는 것보다,
"정확히 어떤 태스크를 가르치고 싶은가"가 더 중요합니다.

예:

- melody+piano -> drums
- chord guide -> piano comp
- full band context -> bass completion
- selected bars -> fill variation

이런 식으로 task를 잘 쪼개면 적은 데이터로도 훨씬 효율이 좋아집니다.

11-2. 적은 데이터에서 효율을 높이는 방법

1) task-specific pair 구축
   한 모델로 모든 음악을 배우게 하기보다, 특정 편집 태스크에 맞는 input-output pair를 만든다.

2) retrieval 사용
   모든 것을 파라미터에만 기억시키려 하지 않고, 비슷한 과거 예시를 찾아 참고하게 한다.

3) rule skeleton + model refinement
   모델이 처음부터 완성본을 다 쓰게 하지 말고,
   규칙 기반으로 skeleton을 만들고 모델은 embellishment/variation만 맡긴다.

4) 사용자 수정 로그를 preference data로 저장
   AI가 만든 뒤 사용자가 어떻게 고쳤는지가 매우 좋은 학습 데이터다.

5) 작은 adapter/LoRA를 사용자별로 둔다
   거대한 본모델 하나를 계속 갈아엎기보다, 공용 base + user adapter가 훨씬 현실적이다.

6) metadata를 많이 저장한다
   악기, 역할, 장르, BPM, 섹션, energy, 설명 태그가 있으면 적은 데이터로도 훨씬 배움이 빠르다.

11-3. 현실적인 데이터 규모 감각

아래는 매우 거친 감각치지만 프로젝트 방향 판단에는 유용합니다.

- 단일 태스크 MVP용: 고품질 200~500개 pair면 시작 가능
- rule/reviewer/retrieval을 같이 쓰면 이보다 적어도 초기 실험 가능
- 사용자 개인화: 잘 라벨된 20~100개 edit 사례만 있어도 retrieval + small adapter로 체감 변화 가능

즉 "몇만 개 없으면 시작 불가"는 아닙니다.
중요한 것은 데이터 개수보다 데이터 구조와 사용 방식입니다.


12. 사용자 데이터 입력/캡처 기능은 어떻게 설계하는 것이 좋은가

사용자는 결국 "내가 작업한 것을 AI가 읽고 점점 내 스타일을 닮아가길" 원합니다.
이를 위해서는 DAW 안의 data capture 기능이 핵심이 됩니다.

12-1. 캡처 버튼의 역할

DAW에 "데이터 캡처" 혹은 유사한 기능을 두고,
사용자가 특정 프로젝트/트랙/구간을 학습 자산으로 저장하게 합니다.

중요한 점은 "자동 수집만"으로 가기보다,
초기에는 사용자가 최소한의 메타정보를 직접 넣도록 하는 편이 정확도와 데이터 가치가 훨씬 높습니다.

12-2. 캡처 시 유저에게 입력받으면 좋은 정보

프로젝트 단위:

- 장르
- BPM
- 박자표
- 곡 섹션 정보 (verse, chorus, bridge 등)
- 무드/에너지

트랙 단위:

- 이 트랙은 MIDI인가 WAV인가
- 악기 계열은 무엇인가
- 역할은 무엇인가
  예: melody / counter melody / accompaniment / bass / drums / pad / strings / fx / lead / arp
- 사람 연주감을 중요시하는 트랙인가
- 메인 트랙인가 보조 트랙인가

구간 단위:

- 선택 bar range
- 이 구간의 역할
  예: intro fill, verse groove, pre-chorus lift, chorus support
- 이 구간에서 중요한 참고 트랙
- 이 구간에서 AI가 해야 할 작업 종류
  예: continue / infill / replace / add layer / simplify / densify

추가 메모:

- 참고 스타일
- 원하는 질감
- 피하고 싶은 요소

12-3. 저장해야 할 실제 데이터

캡처 시 아래를 함께 저장하는 구조가 좋습니다.

- 원본 MIDI 또는 오디오 파일
- 선택된 구간 정보
- 프로젝트 BPM/key/time signature
- 트랙명 / 악기 / 역할 메타데이터
- 주변 트랙 스냅샷
- 사용자가 최종 수정한 결과
- AI가 제안했던 중간안
- 사용자가 채택/거절한 기록

이렇게 해야 나중에 단순 학습용 데이터뿐 아니라:

- retrieval 예시
- preference 학습
- user clone 구축

에 모두 재사용할 수 있습니다.


13. 사용자 "복제본 작곡가"는 어떤 구조로 가야 하는가

이 기능은 단순히 "더 많이 학습하면 언젠가 됨"이 아닙니다.
명확하게 3단계로 나누는 것이 좋습니다.

13-1. 1단계: memory/retrieval 기반 개인화

가장 먼저 할 수 있는 것은 사용자의 과거 작업을 검색해서 참고하는 방식입니다.
이 단계에서는 모델이 사용자의 모든 스타일을 weights에 다 새기지 않아도 됩니다.

예:

- 비슷한 장르/비슷한 BPM/비슷한 역할/비슷한 구간의 과거 예시를 불러옴
- 생성 전에 참고 예시로 넣거나, reviewer가 기준으로 삼음

장점:

- 데이터가 적어도 바로 시작 가능
- 위험이 적음
- 사용자마다 즉시 체감이 있음

13-2. 2단계: user-specific adapter/LoRA

사용자의 충분한 edit history가 쌓이면,
공용 base model 위에 사용자별 adapter/LoRA를 얹는 방식이 좋습니다.

이 구조의 장점:

- 공용 모델은 일반 음악 문법 담당
- 사용자 adapter는 취향/습관/밀도/프레이징 편향 담당
- 업데이트 비용이 낮음
- 사용자마다 분리 관리가 쉬움

13-3. 3단계: workflow-aware agent

가장 멀리 보면 단순 생성기가 아니라,

- 사용자의 반복 습관을 알고
- 자주 쓰는 악기 배치를 알고
- 섹션별 밀도 취향을 알고
- 작업 흐름까지 이해하는

에이전트형 assistant가 됩니다.

즉 "복제본 작곡가"는 단일 모델 하나가 아니라,

- retrieval memory
- user adapter
- task router
- reviewer
- DAW state reader

가 같이 움직이는 구조로 보는 것이 맞습니다.


14. 샘플/wav도 이해하는 방향은 어떻게 가져가야 하는가

사용자는 현대 작업 방식상 MIDI만이 아니라 샘플/wav도 많이 씁니다.
하지만 여기서 중요한 것은 "처음부터 raw audio를 완벽 이해하는 거대 모델"을 목표로 잡지 않는 것입니다.

14-1. 현실적인 첫 단계

초기에는 사용자가 캡처 시 아래를 직접 입력하게 하는 것이 좋습니다.

- 이 트랙은 MIDI / WAV / audio loop / one-shot 중 무엇인가
- 이 샘플의 역할은 무엇인가
  예: kick loop, percussion texture, riser, impact, vocal chop, ambience
- 곡 안에서 이 샘플이 맡는 기능은 무엇인가
  예: groove foundation, transition, texture, ear candy

즉 "샘플 의미 이해"를 처음부터 end-to-end로 추론하기보다,
메타데이터와 UI 입력을 통해 의미를 구조화하는 편이 훨씬 현실적입니다.

14-2. 그 다음 단계

추후에는 audio analysis를 붙일 수 있습니다.

- onset / rhythm density
- spectral centroid / brightness
- loop periodicity
- transient strength
- tonal/noise ratio
- stem class 추정

하지만 이것도 최종 목적은 "음향 공학적 분석"이 아니라,
"이 샘플이 프로젝트 안에서 어떤 역할을 하는지"를 DAW assistant가 더 잘 이해하는 데 있습니다.

14-3. 중요한 점

사용자가 명시적으로 입력하는 role metadata는
적은 데이터 환경에서 매우 강력한 신호입니다.

즉 샘플/wav 지원은 "거대 오디오 모델 먼저"보다,
"캡처 UX + 역할 메타데이터 + 기본 분석기" 조합으로 가는 것이 좋습니다.


15. 자체 모델을 계속 키우는 것이 좋은가, 아니면 일반 LLM을 써야 하는가

결론부터 말하면 둘 중 하나를 고르기보다, 역할을 나눠 쓰는 hybrid가 가장 맞습니다.

15-1. 일반 상용 LLM이 잘하는 것

- 자연어 이해
- 사용자의 의도 파악
- "이 구간에 어떤 작업을 원하는가" 정리
- 캡처 폼 보조 입력
- 메타데이터 정리
- 플래너/라우터 역할
- 과거 작업 요약
- user memory 검색 질의 생성

즉 control plane에는 매우 유용합니다.

15-2. 일반 상용 LLM만으로는 부족한 것

- 정밀한 symbolic MIDI edit
- 일관된 region-only generation
- bar/grid 정합성
- 트랙 역할별 안정적 MIDI 제어
- 저지연 로컬/반로컬 동작
- 사용자 사적 데이터 기반 지속적 개인화

즉 "실제 MIDI 생성 엔진"으로는 불안정하거나 비용/지연/개인화 측면에서 제약이 큽니다.

15-3. 자체 symbolic 모델이 잘하는 것

- MIDI 토큰 수준 제어
- task-specific generation
- low-latency local inference 가능성
- 사용자별 LoRA/adapters 운영
- DAW 내부 편집과 직접 연결

15-4. 추천 구조

- 상용/일반 LLM:
  사용자 명령 이해, planner, metadata, routing, memory query

- 자체 symbolic music model:
  실제 MIDI 생성/편집/infill/track completion

- retrieval/memory layer:
  과거 사용자 데이터, 예시 검색, preference 반영

- reviewer/ranker:
  생성 결과 평가 및 재생성 결정

즉 "일반 LLM vs 자체 모델"의 대결이 아니라,
"일반 LLM은 두뇌의 언어 부분, 자체 모델은 손과 귀의 symbolic 작업 부분"으로 분업하는 것이 맞습니다.


16. 학습은 계속 의미가 있는가

결론은 "예, 의미 있다. 다만 구조 수정과 같이 가야 한다"입니다.

16-1. 학습이 의미 있는 이유

학습을 더 하면 아래는 분명 개선될 가능성이 있습니다.

- 코드 진행을 따르는 경향
- 스타일 일관성
- 프레이징 자연스러움
- 특정 태스크 숙련도
- 특정 사용자 취향 반영

16-2. 그런데 학습만으로 해결되지 않는 이유

현재 남아 있는 큰 문제 중 일부는 구조 문제입니다.

- target range 정보가 generation path에 완전히 안 먹는 문제
- track role conditioning 부족
- decoder default track 문제
- 피아노 물리성/보이싱 제약 부족
- empty-bar rejection loop 부재

이것들은 데이터를 더 넣는 것만으로 완전히 해결되기 어렵습니다.

16-3. 즉 바른 표현

"학습을 더 하면 좋아질 수 있다"는 맞지만,
"학습만 더 하면 제품 목표에 도달한다"는 아닙니다.


17. 현재 로컬 환경에서 학습/테스트는 가능한가

가능합니다.
실제로 로컬 RTX 4080 환경에서 재학습과 테스트를 수행할 수 있음을 확인했습니다.

17-1. 확인된 점

- CUDA 사용 가능
- GPU: RTX 4080
- base training / SFT LoRA training 가능
- 최신 코드 기준 재학습 가능

17-2. 운영상 주의점

Windows 환경에서는 데이터 로더/병렬 작업 관련 권한/안정성 이슈가 있어,
single-worker 우회가 필요했던 경우가 있었습니다.

이를 위해 별도 스크립트로 SFT LoRA를 단일 worker로 도는 우회 경로를 만들었습니다.

관련 파일:
- run_sft_lora_single_worker.py

17-3. 최신 코드 기준 LoRA 관련 주의점

최근 코드에서는 LoRA rank/shape 기대치가 달라질 수 있으므로,
예전 LoRA를 그대로 가져다 쓰면 규격 불일치가 날 수 있습니다.

따라서:

- 최신 코드 기준으로 새 LoRA를 다시 학습
- lora config를 함께 저장
- 추론 시 실제 active_lora가 켜졌는지 확인

이 세 가지가 중요합니다.

17-4. 실제 교훈

예전에는 학습 로그상 LoRA를 잘 만들었어도,
실제 생성 테스트에서는 active_lora가 None이라 결과 평가가 어긋났던 적이 있었습니다.
즉 "학습 성공"과 "생성에 실제 반영됨"은 반드시 따로 확인해야 합니다.


18. 앞으로의 권장 구조

가장 추천하는 구조는 아래와 같은 5층 구조입니다.

18-1. Layer A: DAW Capture / Intent Layer

- 사용자가 선택 구간/타겟 트랙/작업 종류를 명시
- 프로젝트/트랙/구간 메타데이터를 수집
- 자연어 지시와 DAW 상태를 함께 정리

18-2. Layer B: Context Builder

여기서 SongContext를 만든다.
최소 포함 추천 항목:

- key / tempo / time signature
- section map
- chord map
- active tracks
- track roles
- melodic anchor
- groove profile
- density profile
- register budget
- target task
- target range

18-3. Layer C: Task Router

사용자 요청을 아래 같은 태스크로 분기:

- drums_from_melody_piano
- bass_from_chords
- piano_from_chords
- fill_in_selected_bars
- simplify_dense_part
- add_counter_line

18-4. Layer D: Hybrid Generator

추천은 pure generation 하나보다 hybrid입니다.

- 규칙 기반 skeleton 생성
- 모델 기반 refinement / variation / ornamentation

이 방식이 저데이터 환경과 사람다운 결과에 훨씬 유리합니다.

18-5. Layer E: Reviewer / Ranker / Regenerator

검사 항목 예:

- empty bar ratio
- chord tone ratio
- bar continuity
- note density
- register clash
- role consistency
- human-playable voicing

점수가 낮으면 자동 재생성 또는 대체 후보 재랭킹.


19. 단계별 실행 계획표

19-1. Phase 0: 안정화

목표:
- 최신 코드 기준 generation 품질/학습 경로 안정화

해야 할 일:
- active_lora 확인 루틴 고정
- empty-bar rejection 추가
- selected range 생성 결과를 in-place insert/replace하게 수정
- DAW->server request와 실제 generation path 일치시키기
- Track default accomp fallback 문제 줄이기

완료 기준:
- 생성 결과가 선택 구간 안에 안정적으로 들어감
- 긴 empty-bar 결과가 대부분 걸러짐

19-2. Phase 1: MVP 태스크 완성

목표:
- selected-range + target-track 편집 구조를 주요 악기군에 대해 공통으로 성립시키기

우선 지원 목표 악기군:
- drums
- bass
- piano / accompaniment
- strings / pad
- guitar

첫 검증 순서 예:
- 1차: melody + piano -> drums
- 2차: melody + chords -> bass
- 3차: chords/context -> piano comp
- 4차: context -> strings support

해야 할 일:
- task token/route 명확화
- 악기군별 input-output pair 구조 설계
- reviewer를 악기군별로 분기
- DAW에서 selected bars를 직접 교체/삽입
- target track identity 유지 강화를 위한 조건 추가

완료 기준:
- 사용자가 선택한 4~8마디의 target track을 악기군에 따라 바로 받아 수정 가능
- 드럼뿐 아니라 다른 핵심 파트로도 같은 편집 구조가 재사용 가능

19-3. Phase 2: harmonic accompaniment 확장

목표:
- chords -> piano comp
- melody + chords -> bass
- context -> strings support

해야 할 일:
- role metadata 도입
- voicing constraint 강화
- song context 확장

19-4. Phase 3: 사용자 개인화

목표:
- user memory + retrieval
- user-specific adapter/LoRA

해야 할 일:
- DAW data capture UX
- edit history 저장
- user example retrieval
- 개인화 후보 ranking

19-5. Phase 4: sample/audio 이해 확장

목표:
- wav/sample role-aware assistant

해야 할 일:
- 샘플 캡처 메타데이터
- 기본 audio feature 분석
- 프로젝트 내 역할 분류


20. 지금 지수가 바로 봐야 하는 실제 액션 아이템

우선순위대로 적으면 아래입니다.

1) DAW request와 inference server가 task/start_bar/end_bar를 진짜로 반영하도록 연결
2) generated clip append가 아니라 selected range in-place replace/insert로 수정
3) decoder의 default accomp fallback이 어떤 조건에서 발동하는지 줄이기
4) reviewer + regeneration loop에 empty-bar / sparse-output 거르기 추가
5) 제품 단기 범위는 주요 악기군 전체로 유지하되, 첫 번째 검증 태스크만 drums_from_context로 시작
6) task-specific SFT pair 구조를 별도로 설계
7) SongContext 계층을 metadata 수준이 아니라 실제 generation condition으로 승격
8) role metadata를 데이터 구조에 추가
9) user data capture 스키마 설계
10) base model + user adapter 구조를 전제로 개인화 방향 설계


21. 최종 결론

지금 코드베이스는 망한 것이 아니라, 방향 전환이 필요한 상태입니다.
특히 최근 업데이트는 분명 올바른 방향의 수정이 들어갔고, 일부 문제는 실제로 개선되었습니다.

하지만 사용자가 원하는 최종 제품은
"미디 뒤를 이어 쓰는 모델"보다
"DAW 안에서 특정 구간과 특정 트랙을 문맥을 보고 채우는 작곡가용 커서"
에 훨씬 가깝습니다.

따라서 앞으로 중요한 것은:

- 더 많은 generic MIDI를 무작정 넣는 것
- base model loss를 더 낮추는 것

보다도,

- task를 정확히 쪼개고
- DAW 입력과 모델 입력을 맞추고
- role/context/target range를 명시적으로 다루고
- reviewer/regeneration과 hybrid generation을 붙이고
- user data capture를 통해 personalization 루프를 만드는 것

입니다.

한 문장으로 정리하면:

"학습은 계속 의미 있지만, 지금 프로젝트를 진짜 제품으로 만들려면 모델 성능 경쟁보다 편집형 구조와 데이터 구조를 먼저 맞춰야 한다."
