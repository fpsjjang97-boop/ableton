------------------------------------------------------------
1. 실험 개요
------------------------------------------------------------

[1차 학습]
- 데이터: TEST MIDI 폴더 (188곡)
- 증강 후: 2,286파일 / 17,606,526 토큰
- Best Val Loss: 0.0046
- 문제 발견: 폴더 내 수동 전조 파일(+1, +2, -1 등) 다수 포함
  → 특정 곡이 6~7배 중복 학습되는 데이터 오염 발생

[2차 학습]
- 데이터: midi_data 폴더 (40곡, 수동 전조 파일 제거)
- 증강 후: 2,286파일 / 17,606,526 토큰 (동일)
- Best Val Loss: 0.0045
- 총 학습 시간: 0.8시간 (약 47분)


------------------------------------------------------------
2. 청취 리뷰 결과 (1차 / 2차 동일)
------------------------------------------------------------

평가 기준 (각 1~5점):
  (a) 음악으로 인지되는가          → 1/5
  (b) City Pop 장르 느낌이 있는가  → 1/5
  (c) 화성 진행이 자연스러운가     → 1/5
  (d) 8트랙이 독립적으로 들리는가  → 1/5

추가 청취 소감:
- MIDI 노트 이벤트 자체가 거의 없음 (대부분 빈 구간)
- 음이 나오더라도 음악적 맥락 없이 무작위로 배치된 느낌
- 조성이나 리듬 구조가 전혀 느껴지지 않음
- 1차(오염 데이터)와 2차(클린 데이터) 결과가 체감상 동일


------------------------------------------------------------
3. 핵심 의문 — 데이터 문제인가, 구조 문제인가
------------------------------------------------------------

과적합(overfitting)이 데이터 부족의 원인이라면,
모델이 학습 데이터를 "암기"하는 방향으로 수렴하기 때문에
생성 결과가 원본 데이터와 비슷한 결이라도 나와야 정상임.

그러나 실제 출력은:
- MIDI 노트가 거의 없는 빈 파일에 가까움
- 과적합 결과물치고는 원본과 너무 다름
- 단순히 데이터가 부족해서 생기는 증상과 다를 수 있음

→ 구조 또는 inference 파이프라인 버그 가능성을 배제할 수 없음


------------------------------------------------------------
4. 확인 요청 사항 (우선순위 순)
------------------------------------------------------------

[확인 1 — 가장 중요]
학습에 사용한 원본 MIDI 파일을 그대로 inference에 입력했을 때
동일하게 빈 MIDI가 출력되는지 테스트 필요.

  - 빈 파일이 나온다면 → inference 파이프라인 또는 토크나이저
    디코딩 로직에 버그가 있을 가능성이 높음
  - 원본과 비슷한 결과가 나온다면 → 과적합은 맞고,
    데이터 2,000곡 확보 후 재학습이 해결책

[확인 2]
토크나이저가 NOTE_ON / NOTE_OFF / velocity 등
실제 음표 이벤트를 정상적으로 인코딩 및 디코딩하는지 단위 테스트 가능한지?
토큰 → MIDI 변환 과정에서 이벤트가 소실되는 버그가 있을 수 있음.

[확인 3]
inference 시 사용하는 temperature, top_p, min_new_tokens 등
샘플링 파라미터가 적절한지 점검 필요.
파라미터가 잘못 설정되면 silence 토큰만 반복 생성될 수 있음.


------------------------------------------------------------
5. 제안하는 다음 단계
------------------------------------------------------------

Step 1. 위 확인 1~3 테스트 먼저 진행
         → 구조/버그 문제인지 데이터 문제인지 확정

Step 2. 버그가 있다면 수정 후 현재 40곡으로 재생성 테스트
         (데이터 추가 전에 파이프라인이 정상인지 검증)

Step 3. 파이프라인이 정상임이 확인되면 데이터 수집 본격화
         목표: 클린 데이터 최소 200~500곡 (장르 다양성 포함)
         주의: 수동 전조 파일은 반드시 제외, 원본만 midi_data에 포함

Step 4. 200~500곡 기준으로 중간 학습 후 청취 리뷰 반복
         (2,000곡 모을 때까지 기다리지 말고 단계적으로 검증)


------------------------------------------------------------
6. 참고 — Val Loss 해석
------------------------------------------------------------

Val Loss 0.004~0.005 구간은 숫자만 보면 낮아 보이지만,
40곡이라는 극소량 데이터에서 나온 수치이므로
모델이 음악 구조를 이해한 것이 아니라
토큰 패턴을 단순 암기했을 가능성이 있음.

의미 있는 Val Loss 비교는 충분한 데이터(최소 200곡 이상) 확보 후
재학습한 결과를 기준으로 판단하는 것이 적절함.
