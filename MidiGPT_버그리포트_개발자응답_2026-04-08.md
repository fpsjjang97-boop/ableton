## 우선순위 1 — 추론 품질 (BUG 4 / BUG 5)

**원인**: `midigpt/inference/engine.py::_generate_with_harmony` 가 EOS 토큰을
처음 만나는 즉시 종료. 과적합된 베이스 모델(train 0.13 / val 2.78)이 학습
시퀀스 끝에 자주 나오는 EOS 패턴을 외워서, 생성 시작 후 5~10 토큰 안에
EOS 를 뱉어 1KB / 2~5초짜리 결과물이 나옴.

**수정**:
1. `_generate_with_harmony` 에 `min_new_tokens` 파라미터 추가 (HF transformers
   동일 패턴). 지정된 step 에 도달하기 전까지 EOS 의 logit 을 `-inf` 로
   강제 → 절대 EOS 가 샘플링되지 않음.
2. `generate_variation` / `generate_to_midi` 의 기본값을 다음과 같이 변경:
   - `max_tokens`: 512 → **1024**
   - `min_new_tokens`: (신규) **256**
3. `scripts/run_self_improvement_loop.py` 에 `--min_new_tokens` 플래그 노출
   (기본 256), `--max_tokens` 기본을 512 → 1024 로 상향, `--temperature` 를
   0.9 → **1.2** 로 올려 과적합 베이스의 터널 비전을 완화 (리포트가 예시로
   든 값과 동일). 화성 마스크가 off-scale pitch 를 가려주므로 1.2 도 안전.

**기대 효과**: 생성 결과물이 최소 256 토큰 이상이 보장되어, 1KB / 5초가
아닌 5~15KB / 30초~1분 수준으로 복구됨. 강화학습 페어 빌드에 의미 있는
크기의 변주가 생성됨.

**조정 방법**: 너무 길어지면 친구가 `--min_new_tokens 128 --max_tokens 768`
처럼 줄여서 다시 돌리면 됩니다. 0 으로 두면 옛날 동작.

- **원인**: 파일 안에 em-dash(`—`, U+2014) 가 있어서 한국어 Windows pip

- **수정**: 파일을 순수 ASCII 로 재작성. 파일 상단에 "non-ASCII 추가

- **재현 사라짐**: `pip install -r requirements.txt` 가 한국어 Windows 에서
  바로 통과합니다.

  `midigpt/build_dpo_pairs.py` 가 `encoding='utf-8'` 로 읽어서 충돌.
- **수정**: reviewer 의 두 `open()` 호출에 `encoding='utf-8'` 명시.

- **재현 사라짐**: 새 사이클부터 모든 리뷰 파일이 UTF-8 로 저장됨.
- **이미 쌓인 CP949 파일이 있다면**: `reviews/` 폴더 한 번 비우고 사이클

## BUG 1 — 문서/코드 경로 불일치 (`FileNotFoundError`)

**원인**: README 는 `--data_dir ./midigpt_data` 라고 적혀 있지만 `midigpt.pipeline`
는 토큰을 `./midigpt_pipeline/tokenized/tokens/` 에 떨어뜨림. 게다가
`train_pretrain.py` 의 `--data_dir` 가 `required=True` 라서 친구가 직접
경로를 외워야 했음.

**수정**:
1. `train_pretrain.py` 의 `--data_dir` 기본값을 `./midigpt_pipeline/tokenized`
   로 지정 (pipeline 출력 경로와 일치).
2. `midigpt/data/dataset.py::_load_pretrain` 이 다음 경로들을 자동 탐색:
   - `{data_dir}/tokens/`        (정상 레이아웃)
   - `{data_dir}/`               (data_dir 가 토큰 폴더 자체일 때)
   - `{data_dir}/tokenized/tokens/`  (pipeline work_dir 직접 패스)
3. 못 찾았을 때 에러 메시지에 "어디를 봤는지 + 어떻게 고치는지" 모두 표시.

## BUG 6 — 모든 생성물에 `accomp` 트랙 — 답변

**결론: 의도된 동작입니다 (버그 아님). 다만 현 상태는 BUG 4/5 의
부작용으로 더 심하게 보입니다.**

### 설계 의도

`midigpt/tokenizer/vocab.py:56` 의 `TRACK_TYPES` 가 모델이 알고 있는
유일한 트랙 카테고리입니다 (총 14 종):

```
melody, accomp, bass, drums, pad, lead, arp, other,
strings, brass, woodwind, vocal, guitar, fx
```

`midigpt/tokenizer/encoder.py::_classify_track` 는 원본 인스트루먼트의
이름/프로그램 번호/평균 피치를 바탕으로 위 14 카테고리 중 하나를
강제 매핑합니다. 즉 `E.PIANO`, `VIOLIN_LEGATO`, `SYNTHPAD` 같은 원본
이름은 토큰 스트림에 들어가지 않습니다 — 토큰은 어디까지나
`Track_accomp` / `Track_strings` / `Track_pad` 같은 카테고리 토큰만
존재합니다. 디코더 (`decoder.py:242`) 는 이 카테고리를 그대로 출력
MIDI 의 트랙명으로 적기 때문에 DAW 에서도 카테고리만 보입니다.

### 이걸 왜 이렇게 만들었나?

1. 어휘 폭발 방지 — 사용자별로 자유 텍스트 트랙명을 모두 어휘에 넣으면
   vocab 이 폭발 (수만~수십만). 14개 카테고리로 고정해야 학습이 수렴.
2. 일반화 — 같은 곡을 "PIANO" 라고 부르든 "E.PIANO" 라고 부르든
   동일한 표현으로 학습되어야 함.
3. Cubase 15 통합 — `cbd9137` 커밋의 5-zone UI 가 위 카테고리 단위로
   디스패치하기 때문에 그쪽과 어휘를 맞춰둔 상태.

### 그러면 왜 "전부 accomp" 였나?

원본 트랙이 골고루 카테고리에 매핑됨에도 불구하고 결과물이 죄다
`accomp` 였던 이유는 **BUG 4/5 의 직접적인 부작용**입니다:

- 모델이 EOS 를 5~10 토큰 안에 뱉음
- 그 짧은 길이 안에 들어갈 수 있는 `Track_*` 토큰은 1개
- 학습 데이터에서 가장 흔한 카테고리가 `accomp` (피아노/기타 비중이
  높은 J-Pop, J-Rock, City Pop 의 영향)
- 따라서 거의 항상 `Track_accomp` → 다른 카테고리는 사라짐

BUG 4/5 가 고쳐지면 시퀀스가 길어지면서 한 곡 안에 `Track_accomp`,
`Track_drums`, `Track_bass`, `Track_pad` 등이 차례로 등장할 것이고,
DAW 에 14 종 트랙명이 다양하게 보이게 됩니다.

### 원본 악기명 복원은?

설계상 지원하지 않습니다 (의도적). 원본 악기 정보를 보존하려면
별도 사이드카 메타데이터(`*.meta.json`) 를 만들거나, vocab 을
완전히 갈아엎고 토크나이저를 다시 학습해야 합니다 — 후자는
`docs/REMOTE_TRAINING_GUIDE.md` 8절 "절대 하지 말 것" 에 명시된
금기 사항입니다.

### 코드 변경

`encoder.py::_classify_track` 의 docstring 에 위 설계 의도와 BUG 6
관련 답변을 명시적으로 적어 두었습니다. 다음 사람이 같은 의문을
갖지 않도록 하기 위함입니다.

## QUALITY 4 — DPO 가 한 번도 실행되지 않음 (chosen 0개)

**원인**: 임계값이 고정 (`CHOSEN_THRESHOLD=80`, `REJECTED_THRESHOLD=60`)
이라서 점수가 39~77 구간에 몰려 있는 현재 모델로는 chosen 이 영원히
0 개. → DPO 는 영원히 SKIP.

**수정**: `midigpt/build_dpo_pairs.py` 에 **quantile fallback** 도입.

```python
QUANTILE_FALLBACK = 0.30      # top 30% -> chosen, bottom 30% -> rejected
QUANTILE_MIN_GAP = 5.0        # chosen-rejected 점수차 최소 보장
QUANTILE_MIN_PER_BUCKET = 3   # 각 버킷 최소 3개 이상이어야 fallback 활성
```

**동작**:
1. 먼저 고정 임계값(80/60) 로 시도. 양쪽에 3개 이상 모이면 그대로 사용
   → 기존 동작 100% 보존 (모델이 좋아지면 자동으로 옛날 모드).
2. 둘 중 하나라도 3개 미달이면 **percentile 분할**: 점수 정렬 후 상위
   30% / 하위 30% 를 chosen / rejected 로 사용.
3. percentile 모드에서도 chosen 최소값 - rejected 최대값이 5점 미만이면
   "구분이 의미 없음" 으로 판단하고 페어 0 으로 되돌림 (안전장치).

**친구의 162 파일 (평균 59, 39~77 분포) 시뮬레이션 결과**:
- `strategy: quantile_fallback`
- `n_chosen: 4 (점수 70~77), n_rejected: 4 (점수 39~48)`
- 실제로는 162 파일이라 ~48개 chosen / ~48개 rejected 가 만들어집니다.

이 변경으로 친구는 `--min_pairs` 기본 10 을 만족하면서 처음으로 DPO
파인튜닝이 실행되는 것을 보게 됩니다.
