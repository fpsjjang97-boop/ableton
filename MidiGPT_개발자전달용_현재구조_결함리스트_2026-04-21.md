# MidiGPT 현재 구조 결함 리스트

작성일: 2026-04-21

## 목적

이 문서는 해결책 설명을 최소화하고, 현재 MidiGPT 구조에서 드러난 결함만 리스트업한 문서입니다.

## 현재 구조 결함

1. 모델의 기본 학습 목표와 제품 목표가 어긋나 있음

- 현재 프리트레인 목표는 `next-token prediction` 중심이다.
- 하지만 실제 제품 목표는 `작곡가의 커서`처럼 조건을 보고 특정 파트를 생성/보완하는 것이다.
- 즉, 모델이 원래 잘하도록 학습된 일과 지금 요구받는 일이 다르다.

관련 파일:
- [train_pretrain.py](C:/Users/USER/ableton/midigpt/training/train_pretrain.py)

2. 생성 단위가 너무 큼

- 현재는 `코드 진행만 주고 16마디/32마디 전체를 한 번에 생성`하는 식의 사용이 발생하고 있다.
- 그런데 현재 구조는 이런 장거리 조건부 생성보다 `이어쓰기`, `변주`, `부분 보완`에 더 적합하다.
- 결과적으로 사용 방식이 모델의 안정 영역을 벗어나기 쉽다.

3. 화성 제약이 약함

- 현재 harmonic constraint는 사실상 `off-scale note 차단`에 가깝다.
- 즉, chord tone을 강하게 강제하지 않는다.
- 그래서 코드 진행이 입력되어도 실제 출력이 `그 코드로 들리는 반주`가 되지 않을 수 있다.

관련 파일:
- [engine.py](C:/Users/USER/ableton/midigpt/inference/engine.py)

4. chord progression을 구조적으로 고정하지 못함

- 사용자가 원하는 chord progression을 줘도, generation 과정에서 각 마디의 화성 중심이 쉽게 흐려진다.
- 현재 구조는 `주어진 progression을 반드시 따라야 하는 시스템`이라기보다, `그 주변에서 생성하는 시스템`에 가깝다.
- 결과적으로 코드 진행이 텍스트/토큰에는 존재해도 귀에는 잘 안 들릴 수 있다.

5. 스타일 제어 표현력이 부족함

- 현재 vocab 수준에서 스타일 표현이 제한적이다.
- `ballad`처럼 사용자가 실제로 자주 요구하는 스타일이 직접 토큰화되어 있지 않다.
- 따라서 스타일 제어가 정확한 조건 제어라기보다 느슨한 유도 수준에 머문다.

관련 파일:
- [vocab.py](C:/Users/USER/ableton/midigpt/tokenizer/vocab.py)

6. task identity가 모호함

- 현재 입력만 보고는 모델이 `무슨 작업을 하라고` 요청받았는지가 명확하지 않다.
- 예를 들어 아래 태스크들이 구조적으로 구분되어 있지 않다.
  `melody -> accompaniment`
  `chords -> piano comp`
  `missing bass generation`
  `bar infill`
  `variation`
- 즉, task-conditioned generation 구조가 부족하다.

7. target track 조건이 약함

- 사용자가 특정 파트를 원해도, 현재 구조는 그 파트를 정확히 지정하고 생성하게 만드는 장치가 약하다.
- `어떤 트랙을 생성해야 하는지`, `어떤 트랙은 유지해야 하는지`, `어느 구간만 생성해야 하는지`가 명시적으로 분리되어 있지 않다.

8. 전체 곡 수준의 공통 컨텍스트가 없음

- 여러 트랙을 생성해도, 이를 한 곡처럼 묶어주는 상위 SongContext가 부족하다.
- 즉, key / bpm / section / groove / density / motif / energy curve 같은 공통 지휘 정보가 구조적으로 약하다.
- 그래서 각 트랙이 따로 생성되면 서로 통일감이 떨어지기 쉽다.

9. 멀티트랙 생성 순서 전략이 약함

- 현재 구조는 `어떤 트랙을 먼저 만들고, 이후 트랙이 무엇을 참고해야 하는지`가 강하게 설계되어 있지 않다.
- 그 결과 melody, harmony, bass, drums, pad가 서로 종속적으로 정렬되지 않고 느슨하게 생성될 수 있다.

10. reviewer / validator 계층이 약함

- 생성 후 `이 반주가 실제로 코드 진행을 잘 따르는지`, `강박에서 안정적으로 착지하는지`, `트랙끼리 충돌하는지`를 점검하는 구조가 약하다.
- 즉, 결과물을 평가하고 걸러내는 자동 품질 게이트가 부족하다.

관련 후보 파일:
- [reviewer.py](C:/Users/USER/ableton/agents/reviewer.py)

11. long-form coherence 보장이 약함

- 16마디 이상 생성 시 motif, density, register, tension 흐름이 유지되지 않을 수 있다.
- 현재 구조는 긴 구간에서 구조적 반복/대비/발전까지 안정적으로 관리하는 형태는 아니다.

12. 규칙 기반 생성기와 모델 생성기가 분리되어 있음

- 프로젝트 안에는 이미 규칙 기반 chord/arrangement generator가 존재한다.
- 하지만 이것과 MidiGPT 추론이 통합된 단일 생성 경로로 설계되어 있지 않다.
- 그래서 안정적인 skeleton과 유연한 모델 generation의 장점을 동시에 못 쓰고 있다.

관련 파일:
- [composer.py](C:/Users/USER/ableton/agents/composer.py)

13. 현재 구조는 “자율 작곡기”처럼 보이지만 실제로는 “조건부 편집기”에 더 가까움

- 구조 자체가 완전 자율 작곡에 최적화되어 있지 않다.
- 그런데 사용자 기대는 종종 완전 자율 작곡 쪽으로 올라간다.
- 이 간극이 품질 불만으로 바로 나타난다.

## 한 줄 요약

현재 MidiGPT 구조의 핵심 결함은 `조건부 파트 생성/편집기`로 설계되어야 할 시스템을,
실사용에서는 `정밀한 코드 기반 자율 작곡기`처럼 쓰게 되고 있다는 점이다.
