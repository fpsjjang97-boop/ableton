## 평가 방식

각 영역을 다음 5축으로 평가:
- **현황** (지금 상태)
- **강점** (잘한 것)
- **약점** (부족한 것)
- **점수** (10점 만점, 동급 오픈소스/상용 대비 상대 점수)
- **즉시 액션 1-2개**

페르소나별 시각도 같이 제시한다 (사업가 / 작곡가 / 개발자).

## 1. 프로젝트 전반 평가

### 현황

- 데이터: 54곡 원본
- 협업 구조: 작곡가 1 + 개발자 1 + Claude (개발 보조)
- 외부 자금: 0

### 강점
1. **방향성 정확** — DAW 통합 + 자체 MIDI LLM + 화성 마스킹은 모방 어렵고 차별화됨
2. **표준 명세 문서가 충실** — 11개 명세가 코드와 분리되어 있어 협업/감사 가능
3. **속도** — 11일 만에 50M 모델 학습, JUCE 앱, 멀티 에이전트, Audio2MIDI, Sheet2MIDI 통합. 비현실적으로 빠름.
4. **회귀 대응 문화** — 2026-04-08 버그 리포트에 24시간 안에 코드 수정 + 응답 문서 (`MidiGPT_버그리포트_개발자응답_2026-04-08.md`) 작성. 이 사이클이 살아 있는 게 가장 중요.
5. **역할 분리** — 작곡가가 데이터, 개발자가 코드. 의존성 명확.

### 약점
1. **데이터 절대 부족** — 54곡으로 50M 모델은 무리. 가장 큰 병목.
2. **사람 청취 인프라 0** — 룰베이스 리뷰어에만 의존. 음악 평가의 신뢰성 낮음.

4. **사업 문서가 코드보다 늦었음** — 정체성/타겟/수익화가 작성 늦음. 본 작업으로 해소.
5. **외부 검증 부재** — 동업자 2명 외 외부 음악가 검증 없음.

### 점수: **6.5 / 10**
- 같은 단계 오픈소스 음악 LLM 프로젝트 대비 월등히 높음 (구조 + 통합 + 문서 측면)
- 단, 데이터/외부 검증 부재로 "프로토타입" 범주를 못 벗어남
- 이 점수는 향후 30일 내에 8점으로 올릴 수 있음 (데이터 + 청취 패널만 잡으면)

**개발자**: "11일 만에 이만큼 한 건 실력이 아니라 미친 짓이다. 회귀 리스크가 누적되고 있다. 표준 명세 + 사업 문서가 생긴 건 좋고, 이제 회귀 테스트 자동화 + 인코딩/경로 가드 + CI 가 필요하다."

## 2. 제작한 DAW 평가

### 현황
- **JUCE C++ 프론트엔드** (`juce_app/Source/`)
  - Main.cpp 108 lines
  - AIEngine.cpp 1161 lines (LLM 연동 핵심)
  - Core: AudioEngine, MidiEngine, SynthEngine, TrackProcessor, ProjectState
  - UI: MainWindow, SessionView, PianoRoll, MixerPanel, DetailView, FileBrowser, LookAndFeel
- **Python 보조 앱** (`app/core/`)
  - ai_engine, audio_engine, midi_engine, synth_engine, harmony_engine, groove_engine
  - effects_engine, arrangement, automation, score_engine, similarity_engine, prompt_parser
  - 약 18개 모듈, 작은 워크스테이션 형태
- **DAW 통합**
  - Ableton Bridge (`agents/ableton_bridge.py`) ✅
  - Cubase 15 어휘 통합 (vocab.py에 32 articulation, 13 dynamics, CC11/1/64) ✅
  - VST3/CLAP 호스팅: ❌ (계획됨)
  - Logic Pro: ❌

### 강점
1. **JUCE 기반 자체 DAW**는 단순 데모 수준이 아닌 실제 워크스테이션 형태. 1161 라인의 AIEngine은 진지한 통합.
2. **5-zone UI** (커밋 `0083900` 기반) — Cubase 15 워크플로우에 맞춰 디자인
3. **Cubase 15 어휘 통합** (298 articulation 중 32개 핵심, 82 arp pattern, 527 token) — 실연주 표현력의 기반 작업
4. **Ableton Bridge 동작** — 작곡가가 실제로 쓸 수 있는 통합
5. **Python core가 실험/검증용으로 분리됨** — JUCE 빌드 안 해도 모듈 단위 검증 가능

### 약점
1. **VST3/CLAP 호스팅 없음** — 가장 큰 격차. 이게 있어야 진짜 "DAW 안에 사는 LLM" 정체성 완성
2. **시연 가능한 워크플로우 영상 없음** — UI는 있지만 외부에 보여줄 30초 데모 없음
3. **실시간 latency 검증 미흡** — 30초 변주가 실제 5초 안에 나오는지, 사람이 경험한 데이터 없음
4. **macOS / Linux 빌드 미검증** — Windows 위주, 타 OS 회귀 위험
5. **DAW 통합이 "out-of-process"** — VST3 플러그인이 아니라 별도 앱 + Bridge → 작곡가 입장에서 컨텍스트 스위치 비용

### 점수: **7 / 10**
- 자체 LLM 제품 중 DAW 통합이 이 수준인 곳이 거의 없음 (Magenta Studio도 Ableton Pack 한정)
- 단, VST3 호스팅 모듈 부재가 1.5점 깎임. 이게 들어가면 8.5점.

### 페르소나별 코멘트

**사업가**: "JUCE 앱이 있는 게 회사 가치를 두 배로 올려준다. '자체 DAW + 자체 LLM' 패키지로 묶어서 팔 수 있다. 다만 시연 영상 없으면 0과 같다."

**작곡가**: "Ableton Bridge로 일단 사용 가능한 건 좋다. 하지만 진짜 중요한 건 Cubase에서 VST 슬롯에 꽂아서 쓸 수 있느냐다. 그게 안 되면 새 DAW를 또 배우란 소리고, 작곡가는 그걸 안 한다."

**개발자**: "JUCE 앱 빌드 + Python 보조 앱 + Bridge + LLM 에이전트, 이 4개 컴포넌트가 동시에 굴러가는 건 운영 복잡도가 높다. CI/통합 테스트 없이는 회귀 누적된다. VST3 호스팅은 노력 L 짜리고, 데이터 작업과 병렬 진행 가능."

### 즉시 액션
1. Cubase에서 동작하는 30초 시연 영상 1개 (현재 결과로라도 우선 촬영)
2. VST3 호스팅 모듈 프로토타입 시작 (Q3 마일스톤)

## 3. DAW에 탑재한 LLM (MidiGPT) 평가

### 현황
- **모델**: 50M decoder-only Transformer
  - n_layer 12, n_head 12, n_embd 576, n_inner 2304, block_size 2048
  - RoPE (theta 10000), RMSNorm, SwiGLU, weight tying
  - vocab 448 (계층적 REMI: Structure / Harmony / Note / Expressive)
- **추론**: KV cache, FP16, 화성 마스킹, repetition penalty, no-repeat n-gram, num_return_sequences, min_new_tokens=256 (BUG 4/5 fix)
- **학습**: AdamW + cosine warmup, EMA (0.999), bf16/fp16 mixed precision
- **LoRA**: rank 16~32, target 7개 모듈 (q/k/v/o + gate/up/down)
- **현재 학습 상태**: train 0.117 / val 2.755 (54곡, 5.2M 토큰)

### 강점
1. **현대 LLM 표준 아키텍처** — LLaMA/Mistral 계보. RoPE + RMSNorm + SwiGLU 조합은 50M 클래스에서 가장 효율적.
2. **MIDI 도메인 특화 토크나이저** — 448 vocab으로 어휘 폭발 방지하면서 표현력 충분. 14 트랙 카테고리 + 32 articulation + CC + PitchBend.
3. **화성 마스킹** — 추론 시 off-scale pitch 강제 차단. 텍스트 LLM 래퍼는 절대 못 함. **이게 진짜 차별화 포인트**.
4. **Phase 1 안정화 완료** — KV cache, EMA, repetition penalty, no-repeat n-gram, num_return_sequences 모두 적용
5. **인코딩 일관성** — Cubase 15 어휘 통합으로 학습/추론/UI가 같은 어휘를 공유 → 데이터-모델-DAW가 한 줄에 정렬
6. **LoRA 핫스왑** — 스타일 어댑터 교체 가능. 이게 사업적으로 LoRA 마켓 모델의 기반.

### 약점
1. **데이터 결핍으로 인한 과적합** — train/val gap 2.6은 "외운 것 외에는 못 만드는" 상태
2. **EOS 조기 종료 잔존** — min_new_tokens=256로 1KB → 2KB까지 늘렸지만, 학습 자체가 부족해서 8KB+ 결과물이 안 나옴
3. **CFG 학습 미적용** — 스타일 통제력 부재. "재즈로 만들어줘"가 안 통함
4. **Voice Leading 검증 부재** — 자동 평가는 룰베이스, 사람 청취는 패널 부재
5. **Multi-task heads 없음** — chord/key/section을 하나의 head로 학습. 이론 학습이 약함
6. **벤치마크 부재** — 외부 모델 (Music Transformer, MMM, Anticipation) 대비 비교 데이터 없음

### 점수: **6 / 10**
- 아키텍처 단독: 8/10 (산업 표준, 효율적, 도메인 특화)
- 학습 상태: 4/10 (과적합, 데이터 결핍)
- 평균 → 6
- 데이터만 채우면 8점으로 직행 가능

### 페르소나별 코멘트

**사업가**: "50M이라는 숫자가 외부에 잘 팔린다. '대형 LLM 아니어도 도메인 특화로 해결한다'는 내러티브. 단, 결과물이 음악으로 들려야 한다. 지금은 그게 안 된다."

**작곡가**: "구조는 신뢰할 수 있다. 화성 마스킹이 들어가 있다는 건 매우 안심이다. 어떤 변주를 만들어도 절대 어색한 음이 안 나온다는 보장이니까. 다만 들어보면 변주 자체가 비어 있다. 이건 학습 부족이지 구조 문제는 아니다."

**개발자**: "구조는 안 만지면 된다. 데이터 → 사전학습 재실행 → SFT → 사람 페어 DPO 순서로 가면 6점이 8점이 된다. CFG는 데이터 확보 후 즉시 적용. 노력 M, 임팩트 ★★★★★."

### 즉시 액션
1. **데이터 1500곡까지 확보 후 사전학습 재실행** (가장 큰 임팩트)
2. **CFG 학습 적용** (cond drop 10-20%, 데이터 확보 후 즉시)

## 4. 학습 DB 평가

- **목표**: 2000+ 원본곡 (README 명시)
- **자동 품질 필터**: 미구현
- **외부 데이터셋 통합**: Lakh / GiantMIDI-Piano / Slakh2100 모두 미구현
- **사람 라벨 데이터**: 0건

### 강점
1. **장르 다양성** — 적은 곡 수에도 City Pop, J-Rock, Metal, House, K-Pop Garage, Latin, Waltz 등 7+ 장르 커버
2. **고품질 시드** — 작곡가가 직접 만든 곡. 파일명이 곡 정보(BPM, 박자)를 담고 있어 자동 메타 추출 가능
3. **증강 파이프라인** — 전조 + 트랙 드롭아웃으로 x15 증강 (54 → 660). 적은 데이터에서 최대 활용
4. **DATA_GUIDE.md** — 작곡가가 어떤 형식으로 곡을 추가해야 하는지 가이드 존재
5. **GM 매핑** — 드럼 GM 매핑 권장 → 모델이 일관되게 학습 가능

### 약점
1. **곡 수 결정적 부족** — 54 / 2000 = 2.7%. **이게 모든 품질 문제의 근원.**
2. **외부 데이터셋 통합 0** — Lakh/GiantMIDI/Slakh 모두 코드만 명세돼 있고 실제 통합 0
3. **자동 품질 필터 0** — 외부 데이터를 받아도 노이즈 곡을 거를 도구 없음
4. **장르 분포 편향** — City Pop / J-Rock / K-Pop 위주, Classical 비중 낮음
5. **사람 라벨 페어 0건** — DPO 학습은 룰베이스 리뷰어 페어만 존재
6. **저작권 문서 부재** — `SOURCES.md` 같은 출처 추적 문서 없음 (사업적 리스크)
7. **MAESTRO 통합 상태 불명** — README는 93곡 언급하나 midi_data 폴더에 없음. 토크나이즈 통합 여부 확인 필요

### 점수: **3 / 10**
- 시드 데이터 품질만 8/10
- 양 0/10
- 외부 통합 0/10
- 사람 라벨 0/10
- 평균 → 3
- **이 점수가 프로젝트 전체의 병목.** 7점까지 올려야 함.

### 페르소나별 코멘트

**사업가**: "데이터가 없으면 모델이 없고 모델이 없으면 사업이 없다. 자금 우선순위 1번. 외부 데이터셋 라이선스 검토 + 동업자 작곡 페이스 가속 + 사람 라벨 인프라. 셋 다 동시에."
