`midi_data/` 폴더에 **수동 transposed 파일 14개**가 원본과 섞여 있어, `pipeline.py`의 자동 augmentation과 **이중 적용**되고 있습니다. 그 결과 일부 곡(HOUSE / METAL / J ROCK 등)이 정상보다 **약 6.4배** 학습 비중을 차지하는 분포 편향이 발생했습니다.

## 1. 오염 진단

### 1-1. 발견된 수동 transposed 파일 (14개)

| 파일 | 변형 | 원본 |
|------|------|------|
| `HIPHOP 63 4-4 ALL +1.mid` | +1 semitone | `HIPHOP 63 4-4 ALL.mid` |
| `HOUSE 123 4-4 ALL +1.mid` | +1 | `HOUSE 123 4-4 ALL.mid` |
| `HOUSE 123 4-4 ALL +2.mid` | +2 | 동일 |
| `HOUSE 123 4-4 ALL +3.mid` | +3 | 동일 |
| `HOUSE 123 4-4 ALL -1.mid` | -1 | 동일 |
| `J ROCK 130 4-4 +1.mid` | +1 | `J ROCK 130 4-4.mid` |
| `J ROCK 130 4-4 +2.mid` | +2 | 동일 |
| `J ROCK 130 4-4 -1.mid` | -1 | 동일 |

| `METAL 110 4-4 -1.mid` | -1 | 동일 |

→ **14개 / 54개 = 약 26% 오염**

### 1-2. 이중 augmentation 메커니즘

`midigpt/pipeline.py:55` 기본값이 `--transpose all` (=+1~+11) 이므로:

```
원본 X (1개)
  ↓ 수동 변형 (이미 폴더에 있음)
X, X+1, X+2, X+3, X-1, X-2, X-3 (7개)
  ↓ pipeline 자동 augmentation (×12)
X×12, (X+1)×12, ..., (X-3)×12 = 84개
```

비교:
- **수동 변형 있는 곡** (예: HOUSE 123 4-4 ALL): 7 × 12 = **84개 토큰화 사본**
- **수동 변형 없는 곡** (예: WALTZ 6-8 85 BPM FULL): 1 × 12 = **12개 토큰화 사본**

→ **6.4배 분포 편향**. 모델은 HOUSE/METAL 같은 곡을 다른 곡보다 6.4배 자주 봄.

### 1-3. 직접 영향 (BUG 6 / Quality 1과의 관계)

- **BUG 6 "전부 accomp"** — accomp 카테고리가 가장 흔한 트랙이라 EOS 조기 종료 시 그것만 나옴 (이미 응답 문서에서 분석됨)
- **Quality 1 "장르 미반영"** — 모델이 City Pop / J-Rock / Metal에 편향, 다른 장르를 일반화 못함. **본 오염이 직접 원인**
- **Quality 1의 점수 분포 편향** — METAL 39-48점, WALTZ/J-POP 65-77점. 정확히 augmented된 곡(METAL은 변형 4개)이 낮고 안 된 곡이 높음. 모델이 변형 곡에 과적합되어 새 곡 생성을 못 하고 있음을 시사.

### Step 5 — 효과 검증

학습 종료 후 train_pretrain.py 출력의 마지막 줄을 확인:

| 지표 | 이전 (오염) | 회수 후 목표 | 의미 |
|------|------------|-------------|------|
| Train Loss | 0.117 | 0.3 ~ 0.7 | 너무 낮으면 여전히 과적합 |
| Val Loss | 2.755 | 1.5 ~ 2.0 | val이 떨어지는 것이 회수 효과 |
| Train/Val Gap | 2.638 | 1.0 ~ 1.5 | 1.5 미만이 회수 성공 |

**해석**:
- Train Loss가 **올라가는 것이 정상** (외울 게 줄었음)
- Val Loss가 **떨어지는 것이 회수 효과** (일반화 시작)
- Gap 1.5 미만이면 데이터 540 → 1500 로 다음 단계 진행
- Gap 2.0+ 유지면 오염 외 다른 원인 (데이터 절대 부족) → 무조건 추가 데이터 수집

## 3. 향후 데이터 추가 시 규칙 (재오염 방지)

### 규칙 ①: midi_data/에는 절대 수동 transposed 파일을 넣지 않는다
- 작곡가가 **새 원본 곡만** 추가
- transposition은 pipeline 자동 처리에 맡김

### 규칙 ②: 파일명에 `+`, `-` 숫자가 들어간 파일은 자동 거부
- pipeline.py 에 가드 추가 권장 (다음 코드 변경 제안)

```python
# midigpt/pipeline.py 또는 augment_dataset.py 추가
import re

def is_manually_transposed(filename: str) -> bool:
    """Detect manually-transposed files by name pattern (+1, -2, +3, etc)."""
    # Match " +N" or " -N" or "+N" before .mid (avoid false positives)
    return bool(re.search(r"[+\-]\d+(?=\.mid$|\s|$)", filename))

# pipeline 시작 시 점검
manual_files = [f for f in midi_files if is_manually_transposed(f.name)]
if manual_files:
    print(f"WARNING: {len(manual_files)} manually transposed files detected:")
    for f in manual_files:
        print(f"  - {f.name}")
    print("These will skew distribution. Move them out of midi_data/.")
    sys.exit(1)
```

### 규칙 ③: 새 데이터 추가 시 출처 기록
- `midi_data/SOURCES.md` 파일 만들고, 모든 곡의 출처/작곡자/라이선스 기록
- 동업자 자작곡 / 라이선스 free 데이터셋만 추가

### 규칙 ④: 외부 데이터셋 통합 시 자동 품질 필터 통과 필수
- Lakh, GiantMIDI, Slakh — 모두 필터 적용 후 통합
- 필터 기준: 트랙수≥4, 박자 일관성, 노트 밀도, 키 일관성
