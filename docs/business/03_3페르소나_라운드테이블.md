## 0. 문서의 목적

본 문서는 MidiGPT 프로젝트의 핵심 결정을 **사업가 / 작곡가 / 개발자** 세 페르소나의 대화 형식으로 정합화한다. 같은 결정을 세 시각에서 동시에 검증함으로써 한쪽이 비대해지는 것을 막는다.

각 페르소나의 발언 규칙:
- **사업가**는 시장/수익/타이밍/정체성을 본다
- **작곡가**는 결과물의 음악성/워크플로우/실제 사용 가능성을 본다
- **개발자**는 기술적 가능성/시간 비용/리스크를 본다
- **세 명 모두 동의하지 않은 결정은 본 문서에 담지 않는다**

## 라운드 1 — 현재 상태 진단

> 2026-04-08 2차 버그 리포트 직후, 자가학습 결과가 실질적으로 음악이 아닌 상황을 보고

**작곡가**: 솔직히 말하면 지금 나오는 결과물은 음악이 아니에요. DAW에서 열어봐도 노트가 몇 개 흩어져 있는 수준입니다. 학습 데이터로 넣은 City Pop이나 J-Rock의 흔적이 전혀 없어요. 1차 패치 후에도 2KB짜리 파일이라 그냥 평가 자체가 불가능합니다.

**개발자**: 인정합니다. 원인은 명확해요. 모델이 50M 파라미터인데 학습 데이터가 54곡밖에 없어서 곡당 92만 파라미터가 할당되는 셈입니다. 이건 모델이 데이터를 일반화하는 게 아니라 외우는 영역이에요. 외운 결과로 EOS 토큰을 학습 시 시퀀스 끝에 항상 본 패턴 그대로 5~10토큰 안에 뱉어버려서 1KB 결과물이 나오는 거예요. min_new_tokens=256을 강제로 박아서 2KB까진 나오게 했는데, 이건 증상 완화이지 근본 해결은 아닙니다.

**사업가**: 잠깐, 그러면 이게 "버그"인가요 아니면 "정상"인가요? 투자자한테 설명할 때 이게 굉장히 중요해요.

**개발자**: **정상입니다.** 데이터 결핍 상태에서 50M 모델이 보일 수 있는 가장 흔한 증상이에요. train loss 0.117인데 val loss 2.755인 거 보면 명백한 과적합. 50M 모델을 제대로 학습시키려면 보통 수억 토큰이 필요한데 우리는 5.2M 토큰밖에 못 줬으니까 1~2% 수준이죠.

**작곡가**: 그러면 데이터를 더 모으면 되는 거 아니에요?

**개발자**: 네. 그게 제일 큰 병목이고, 사실상 유일한 해결책이에요. 다른 모든 처방은 보조입니다.

**사업가**: 좋아요. 그러면 결론은 "이건 정상이고, 데이터 수집이 정답이다." 맞나요?

**작곡가/개발자**: 동의.

> **Round 1 결론**: 현재의 자가학습 결과 부진은 **데이터 결핍에 의한 정상적인 과적합 증상**이며, 코드 버그가 아니다. 이 진단은 사업가의 외부 커뮤니케이션 자료로 그대로 사용 가능하다.

## 라운드 2 — 구조 변경 vs 학습 지속

> 동업자 리뷰를 받고 "구조를 갈아엎어야 하나, 아니면 학습을 더 해야 하나" 결정해야 함

**사업가**: 동업자 리뷰가 두 가지 방향을 동시에 지적했어요. 하나는 "이 LLM 구조가 음악에 안 맞는 거 아니냐"는 회의론, 다른 하나는 "그냥 학습이 부족한 거 아니냐"는 옹호론. 우리는 어디에 베팅해야 합니까?

**개발자**: 구조는 안 바꾸는 게 맞습니다. 우리가 쓰는 50M decoder-only + RoPE + RMSNorm + SwiGLU + KV cache는 LLaMA/Mistral 계열의 표준이고, MIDI 도메인에 맞게 토크나이저(448 vocab, 14 트랙 카테고리, Cubase 15 어휘 통합)를 잘 짜놨어요. 화성 마스킹도 추론 시 off-scale pitch를 강제로 차단하니까 음악 이론은 무조건 지켜집니다. 이건 텍스트 LLM 래퍼는 절대 따라할 수 없는 부분이에요.

**작곡가**: 음악적으로 봐도 구조 자체는 맞는 것 같아요. 화성 마스킹 같은 건 작곡가 입장에서 정말 환영할 만한 거예요. 코드 진행 위에 절대 어색한 음을 안 낸다는 보장이니까. 문제는 그 보장이 있어도 **음악적 내용**이 비어 있다는 거고, 그건 학습 부족 문제죠.

**사업가**: 그러면 자가학습 강화는요? 지금 self-improvement loop가 돌아가고 있잖아요. 이걸로 부족분을 메울 순 없나요?

**개발자**: 그게 사실 가장 위험한 함정이에요. 우리 리뷰어가 룰베이스라서 D major를 가정하고 벨로시티 60-100을 좋아해요. 이 룰에 모델이 과적합되면 진짜 음악성은 더 떨어지고 룰 점수만 올라가는 distribution narrow 현상이 옵니다. 우리 로드맵 문서에도 "자기증류 비율 50% 넘으면 distribution 좁아짐, 합성 데이터 30% 넘으면 model collapse" 라고 명시해놨어요.

**작곡가**: 아 그건 진짜 위험하네요. 룰이 바뀌면 음악도 망가지고, 룰을 안 바꾸면 모든 곡이 비슷해지고.

**사업가**: 그럼 자가학습 자체를 폐기?

**개발자**: 아뇨. 자가학습은 **base 모델이 어느 정도 의미 있는 출력을 낼 수 있을 때** 의미가 있어요. 지금은 base 모델이 그 수준이 아닙니다. 그러니 순서가:
1. 데이터 540곡 → 2000+곡
2. base 모델 사전학습 재실행, train/val gap 1.0 미만으로 떨어지는지 확인
3. SFT (원본↔변주 페어, LoRA)
4. **사람이 청취한 페어**로만 DPO. 자동 리뷰어 페어는 보조 50% 미만
5. 자동 사이클은 사람 청취 사이에 3-5회로 제한

**작곡가**: 4번 중요해요. 사람 청취 페어 인프라가 없으면 우리는 영원히 룰베이스의 노예예요.

**사업가**: 좋아요. 그러면 결정:
- 구조 변경 ❌
- 자가학습만으로 해결 ❌
- **데이터 수집 + 사람 청취 페어 인프라 구축 + 그 위에 SFT/DPO** ⭕
- 동의?

**개발자/작곡가**: 동의.

> **Round 2 결론**: 구조는 유지한다. 자가학습은 base 모델이 충분히 학습된 이후 보조적으로만 사용한다. **사람 청취 페어 인프라를 사이클 자동화보다 먼저 만든다.**

## 라운드 3 — 더 나아가는 방법

> 방향성 자체는 옳다는 합의 위에서, 다음 단계 액션을 정한다

**사업가**: 방향성이 맞다는 데 셋이 동의했으니, 이제 어떻게 더 나아갈지 봅시다. 4개 축으로 정리해 본 게 있는데 보고 의견 주세요.

**(축 A: 데이터)**

**개발자**: 가장 임팩트 큰 축이에요. 현재 540곡 → 목표 2000+곡. 출처는:
- **Lakh MIDI Clean (LMD)** 약 17만 곡 → 품질 필터(트랙수, 박자, 노트 밀도, 키 일관성) 후 5~10만 곡 확보 가능
- **GiantMIDI-Piano** 약 1만 곡 (피아노 클래식, 라이선스 OK)
- **Slakh2100** 2100곡 (멀티 stem)
- **sheet2midi 배치 변환** — IMSLP PDF 자동 변환

**작곡가**: 그런데 외부 데이터가 들어오면 우리 동업자 곡들이 희석되지 않아요? 우리 정체성이 City Pop, J-Rock인데 클래식 1만 곡이 들어오면 모델이 클래식을 더 많이 배워버릴 거 같은데.

**개발자**: 좋은 지적이에요. 그래서 두 가지 처방:
1. **Continued pretraining** 방식으로 — 큰 외부 데이터로 base를 먼저 학습하고, 그 위에 우리 동업자 데이터로 낮은 LR로 지속 학습
2. **장르 LoRA** — 베이스는 다양하게, 스타일은 LoRA로 분리. 사용자가 LoRA를 갈아끼우게.

**사업가**: 좋아요. 1번이 사실은 사업적으로도 의미가 있는데, "우리는 큰 데이터로 일반화는 했고, LoRA로 우리만의 색을 입혔다" 라고 설명할 수 있거든요.

**(축 B: 학습)**

**개발자**: 데이터 확보 후 Phase 3로 가는 거예요.
- **CFG 학습** (cond drop 10-20%) — 추론 시 스타일/장르 통제력 비약적 상승. 임팩트 ★★★★★, 노력 M.
- **Multi-task heads** — chord/key/section 동시 예측 → 이론 이해도 ↑
- **Voice Leading reward + GRPO** — 음악 이론 reward 모델

**작곡가**: CFG가 핵심일 것 같아요. 작곡가 입장에서 "이건 jazz 풍으로", "이건 City Pop 풍으로" 라고 통제할 수 있는 게 변주 도구의 핵심이에요.

**(축 C: 정체성)**

**사업가**: 이게 사실 제일 중요한 축이에요. 우리 차별화 포인트를 한 줄로 정의하면?

**작곡가**: "DAW 안에서 일하는 음악 이론을 아는 AI 동료". DAW를 떠나는 순간 작곡가 입장에서 매력 0이에요. 새 도구를 또 배우고 싶지 않거든요.

**사업가**: 그럼 VST3/CLAP 호스팅 모듈이 진짜 차별화네요. 단순 변주 도구가 아니라 DAW 안에 사는 LLM.

**개발자**: 동의. 이건 노력 L 짜리지만 임팩트 ★★★★ 입니다. Phase 1 후속에 들어가 있고, 데이터 작업과 병렬로 갈 수 있어요. 다만 Cubase/Ableton 둘 다 지원하려면 시간이 더 들어갑니다. 우선순위는 Cubase 15(이미 어휘 통합 완료) → Ableton(이미 Bridge 있음) → Logic 순.

**(축 D: 검증)**

**작곡가**: 룰베이스 리뷰어를 사람 청취 데이터로 보정해야 해요. 외부 작곡가 5-10명 패널 만들어서 A/B 청취 테스트 인프라 구축. DPO 페어는 사람 라벨 50% + 자동 50% 비율.

**사업가**: 그 패널은 어떻게 모집하죠? 무보수면 안 모일 텐데.

**작곡가**: LoRA 마켓플레이스의 수수료 share. 자기가 만든 LoRA가 팔리면 70%는 자기 거. 30%는 우리. 패널 참여자들에게 우선권 줘서 초기 라이브러리 채우게 하면 동기부여가 됨.

**사업가**: 좋아요. 그건 Phase C 비즈니스 모델이랑도 정확히 맞물리네요.

> **Round 3 결론**: 4개 축(데이터/학습/정체성/검증)을 동시에 굴린다. 가장 임팩트 큰 건 **데이터**, 가장 차별화 되는 건 **VST3/CLAP 호스팅 + 사람 청취 패널**. 구조 변경/RL 가속은 다음 분기 이후로 미룬다.

## 라운드 4 — 사업 정합성

> 기술 방향이 정해졌으니, 사업 단의 결정을 맞춘다

**사업가**: 이제 사업 단의 질문 몇 개. 첫째, 우리는 누구한테 파는 거죠?

**작곡가**: 1차는 저 같은 세미프로 작곡가. Cubase, Ableton, Logic 쓰고 월 1회 이상 곡 작업하고 막힐 때 영감을 원하는 사람. 2차는 BGM 양산 작곡가. 게임/광고/유튜브.

**사업가**: 두 번째, 수익화. 언제 어떻게?

**개발자**: Phase B (퍼블릭 베타) 까지는 무료. Phase C에 Pro 구독 $15/월, LoRA 마켓 30% 수수료, B2B API.

**작곡가**: Pro 구독 가격은 합리적이에요. Splice가 $13, Output Arcade가 $10인데 우리는 LLM 통합이라 $15는 받을 수 있어요. 다만 무료 티어가 충분히 매력적이어야 함.

**사업가**: 셋째, 출시 시점. 사업적으로는 빨리 내고 싶은데, 작곡가 입장에서는 품질 기준이 어떻게 되죠?

**작곡가**: **외부 작곡가 5명이 30분 이상 청취해서 평균 7/10점 이상 줄 때**. 그 전에 출시하면 첫인상이 나빠서 영원히 회복 못 해요.

**개발자**: 기술적으로 그 조건은 데이터 1500곡 + train/val gap < 1.0 + 평균 MIDI 8KB 이상이 충족되어야 가능할 거예요.

**사업가**: 그러면 출시 = 데이터 + 품질 둘 다 만족할 때. 동의.

**사업가**: 넷째, 정체성 한 줄. 우리가 매일 아침 외워야 할 한 줄.

**작곡가**: "DAW 안에서 작곡가와 같이 호흡하는, 음악 이론을 아는 AI 동료."

**사업가**: 좋아요. 이걸 사업기획서 1.1 절에 그대로 박을게요.

> **Round 4 결론**: 1차 타겟 = 세미프로 작곡가. 수익화 = Phase C부터 Pro 구독 + LoRA 마켓. 출시 조건 = 외부 패널 평균 7/10. 정체성 = "DAW 안에서 작곡가와 같이 호흡하는 음악 이론을 아는 AI 동료".
