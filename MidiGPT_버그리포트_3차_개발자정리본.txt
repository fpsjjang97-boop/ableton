MidiGPT 3차 버그 리포트 (개발자 전달용 정리본)

문서 목적
- 작곡가 측 청취 평가와 실행 로그를 바탕으로, 현재 파이프라인의 핵심 문제를 개발자가 빠르게 파악할 수 있도록 정리한 문서이다.
- 원본 리포트의 핵심만 남기고, 개발 액션 중심으로 재구성했다.

1. 최종 생성 결과 청취 평가
대상 파일
- test_gen.mid

평가 방식
- Cubase에서 직접 열어 청취함.
- 아래 4개 항목을 1~5점으로 평가함.
  1) 음악으로 인지되는가
  2) 입력 장르(City Pop)가 느껴지는가
  3) 화성이 어색하지 않은가
  4) 8개 트랙(bass / accomp / pad / drums / melody / guitar / strings / brass)이 독립적으로 들리는가

평가 결과
- 4개 항목 모두 1점

청취 기반 결론
- 결과물이 "음악"으로 느껴지지 않음.
- 미디 노트가 음악적 문장이나 구조 없이 나열된 느낌이 강함.
- 장르의 특징이 거의 느껴지지 않음.
- 화성 진행의 자연스러움이 부족함.
- 트랙 간 역할 분리가 잘 들리지 않음.
- 이전부터 지적한 accomp 관련 문제가 여전히 남아 있음.

해석
- 현재 문제는 단순 샘플 1개의 실패라기보다, 학습 결과물의 전반적인 품질 문제로 보임.
- 특히 학습 데이터 자체 품질, 데이터 구성 방식, 토크나이징/학습 목표 설계, 트랙 구조 보존 여부를 함께 점검할 필요가 있음.

2. 개발 측에서 우선 확인이 필요한 항목
A. 학습 데이터 품질 점검
- 실제 학습에 사용된 MIDI들을 개발자와 작곡가가 함께 직접 열어 확인할 필요가 있음.
- 확인 포인트:
  - 트랙 분리가 실제로 의미 있게 되어 있는지
  - 장르명이 붙은 데이터가 실제로 그 장르처럼 들리는지
  - 화성/리듬/구성이 음악적으로 성립하는지
  - 같은 곡의 변형본이 과도하게 많아 데이터 다양성이 무너진 것은 아닌지
  - 비정상 MIDI(트랙 누락, 이벤트 꼬임, 구조 붕괴)가 포함되어 있지 않은지

B. accomp 문제 재확인
- 이전 리뷰에서 이미 언급한 accomp 관련 문제가 이번 결과물에서도 계속 보임.
- 기존 수정 사항이 실제 코드/학습 결과에 반영되었는지 재점검 필요.
- 작곡가의 청취 기준과 개발자의 내부 평가 기준이 다를 수 있으므로, 동일 파일을 함께 열어 어떤 현상을 문제로 보는지 기준을 맞출 필요가 있음.

C. 작곡가 측 사용 환경 점검
- 원본 리포트 기준으로는 이전 리뷰 사항이 그대로 남아 있음.
- 다만 일부 문제는 모델 자체가 아니라 작곡가의 학습/생성 환경 문제일 가능성도 있으므로, 재현 환경이 정확히 동일한지 확인이 필요함.
- 확인 포인트:
  - 실제 로드된 체크포인트가 의도한 파일이 맞는지
  - 생성 시 사용된 옵션(style/key/tempo/token 길이 등)이 기대와 일치하는지
  - 후처리/디코딩 과정에서 트랙 정보가 깨지지 않는지

3. 실행 로그 기준 확인된 기술 이슈
3-1. 환경 세팅 자체는 큰 문제 없이 진행됨
확인된 내용
- Python 3.11.9 정상
- PyTorch CUDA 12.1 계열 설치 완료
- RTX 4080 인식 정상
- 주요 프로젝트 import 정상
- 기본 모델 인스턴스 생성 정상
- MIDI 데이터 폴더 인식 정상

주의 사항
- setup_check.py에서 mido 항목은 "version unknown" 경고가 있었음
- 즉시 치명적 오류로 보이진 않지만, 환경이 완전히 깔끔한 상태는 아님

3-2. 데이터 증강은 완료됨
로그상 결과
- 원본 188개 MIDI 기준
- 2098개 augmented 파일 생성
- 총 2286개 파일 구성
- 에러 0

주의 사항
- 증강 수 자체는 많지만, 이것이 곧 데이터 품질 향상을 의미하지는 않음
- 오히려 원본 품질이 낮거나 데이터 다양성이 부족하면, 같은 문제를 대량 복제했을 가능성도 있음

3-3. DPO fine-tune 단계가 실패함
실패 지점
- Cycle 3 - STEP 4: DPO fine-tune

에러 내용
- FileNotFoundError: DPO directory not found: C:\Users\USER\ableton\midigpt\dpo_pairs\dpo

의미
- DPO pair summary는 생성되었지만, 실제 DPO 학습 단계에서 기대하는 디렉토리 구조와 저장 위치가 맞지 않아 파이프라인이 중단됨.
- 즉, self-improvement loop가 끝까지 정상 완료되지 못함.

개발 확인 포인트
- train_dpo.py가 기대하는 데이터 폴더 구조를 다시 확인할 것
- dpo_pairs 저장 위치와 실제 loader가 찾는 하위 경로(dpo)가 왜 불일치하는지 확인할 것
- pair builder와 train_dpo 사이의 경로 계약(contract)이 깨져 있는지 확인할 것
- Windows 경로 기준에서만 발생하는 문제인지도 확인할 것

4. 추가로 의심되는 로직 문제
원본 리포트 메모
- "모든 키와 템포가 40, C키로 잡히는 점이 이상함"

해석
- 생성/분석/리뷰 단계 중 일부에서 key 및 tempo 추정 로직이 비정상적으로 고정되는 현상이 의심됨
- 만약 사실이라면, 메타 정보 인코딩 또는 조건값 반영이 거의 되지 않고 있을 가능성이 있음

개발 확인 포인트
- 입력 MIDI의 실제 tempo/key를 어떤 시점에 추출하는지
- SongMeta 또는 tokenizer/encoder 단계에서 tempo/key가 정상 반영되는지
- 생성 시 조건값(style/key/tempo)이 모델 입력으로 실제 들어가는지
- 후속 평가기(rule-based reviewer)가 잘못된 메타를 기준으로 점수화하고 있지 않은지

5. 현재 단계의 종합 판단
- 현재 결과물은 "학습이 끝났으니 품질만 조금 다듬으면 되는 상태"로 보기 어려움.
- 품질 문제와 파이프라인 문제(DPO 실패)가 동시에 존재함.
- 우선순위는 아래와 같음.

우선순위 1
- DPO 파이프라인 경로 오류 수정
- self-improvement loop가 실제로 끝까지 도는지 확인

우선순위 2
- 학습 데이터 원본을 직접 청취/검수
- 장르명, 트랙 구성, 화성 품질, 구조적 완성도 확인

우선순위 3
- accomp 문제 포함, 이전 리뷰에서 지적된 사항이 이번 빌드에 반영되었는지 대조

우선순위 4
- key/tempo 추정 또는 조건 반영 로직 점검

6. 개발자에게 요청하는 액션 아이템
- test_gen.mid를 작곡가와 동일한 환경 기준으로 직접 열어 문제를 함께 확인해줄 것
- 학습에 사용된 대표 MIDI 몇 개를 같이 열어 데이터 품질을 점검할 것
- self-improvement loop의 DPO 단계 경로 오류를 수정할 것
- key/tempo가 비정상적으로 고정되는 현상이 실제 재현되는지 확인할 것
- 이전 리뷰에서 지적된 accomp 문제와 기타 수정사항이 왜 반영되지 않았는지 확인할 것

7. 한 줄 결론
- 현재 문제의 핵심은 "결과물 품질 저하"와 "파이프라인 미완주(DPO 실패)"이며, 단순 파라미터 조정 이전에 데이터 품질, 경로 구조, 메타 반영 로직을 먼저 점검해야 한다.
