# Skill — 변경 착수 전 체크리스트

`role-design-composer` 가 설계서를 넘기기 전, 그리고 `role-main-coder` 가 편집을 시작하기 전에 공통으로 통과해야 하는 빠른 체크리스트.

## 사용 시점

- 버그 리포트를 받았을 때 → 이 체크리스트를 먼저 돌린다
- "한 줄만 바꾸면 될 것 같다" 는 유혹이 들 때 → 반드시 돌린다
- 리팩토링 / 이름 바꾸기 → 돌린다

## 체크리스트

### 1. 근거 확인
- [ ] 이 변경의 트리거는 무엇인가? (리포트 번호 / 라인 / 에러 메시지)
- [ ] 재현 가능한 증상인가, 추정인가? 추정이면 먼저 재현을 시도.
- [ ] 표면적 원인과 근본 원인이 같다고 확신하는가?
  - 예: 5차 Bug 1 의 표면 원인은 "키 이름 불일치", 근본 원인은 "summary.json 이 glob 에 섞임"
  - 표면만 고치면 같은 패턴이 다른 곳에서 재발

### 2. 영향 범위
- [ ] 변경 파일이 어떤 계약(`rules/01-contracts.md`)을 만족해야 하는가?
- [ ] 같은 계약을 공유하는 다른 파일은? (경계 지점을 전부 열거)
- [ ] 이 변경이 체크포인트/토큰 분포에 영향을 주는가? → BREAKING 표기 예약

### 3. 과거 패턴 대조
- [ ] `rules/05-bug-history.md` A~H 중 이 변경이 건드리는 패턴은?
- [ ] 관련 패턴의 "회귀 방지 체크" 를 만족할 계획이 있는가?

### 4. Fallback 정책
- [ ] 이 영역이 학습 / 추론 / 사용자입력 중 무엇인가? → `rules/02-fallback-policy.md` 의 해당 모드 참조
- [ ] "미지정 값" 을 "명시적 값" 으로 취급하려는 유혹이 없는가?

### 5. 경계 조건
- [ ] OS: Windows + Linux 양쪽에서 돌아야 하는가? (`rules/03-windows-compat.md`)
- [ ] 인코딩 / 경로 / 개행 이슈가 있는가?
- [ ] dtype / device / amp 가 관련되는가?

### 6. 작업 분할
- [ ] Main Coder / Sub Coder 작업이 나뉘는가?
- [ ] 테스트 / 마이그레이션 / 로깅 중 필요한 것은?

## 통과 조건

위 6개 항목 모두에 구체적 답을 적을 수 있을 때 착수. 한 항목이라도 "모름" 이면 먼저 조사 / 사용자 질문.

## 안티 패턴 조기 발견

다음 중 하나라도 해당하면 **설계를 다시 한다**:

- "일단 되게 하고 보자"
- "이건 예외 케이스니까 try/except 로 감싸자"
- "다른 사람이 쓸 땐 기본값으로 두자"
- "이 변경은 너무 간단해서 설계 불필요"

이 문장들이 나오는 순간 패턴 G (광범위 try/except) 또는 패턴 B (기본값으로 에러 삼킴) 의 씨앗.
