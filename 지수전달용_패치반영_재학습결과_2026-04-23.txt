[4] 이번에 적용한 코드 패치

1. dataset.py 수정

- 기존에는 sft_*.json만 읽고 task_*.json은 무시하고 있었다.
- 수정 후에는 sft_*.json + task_*.json 둘 다 읽는다.
- 따라서 task pair가 실제 학습에 들어간다.

2. build_task_pairs.py 수정

- 긴 곡은 통째로 pair를 만들다 버려지는 경우가 많았다.
- 수정 후에는 bar window 방식으로 잘라서 task pair를 생성한다.
- 특히 strings/guitar/piano/drums 같은 역할 기반 task에서 usable pair가 늘어나도록 바꿨다.

3. constrained.py 수정

- bar coverage를 Pitch token 기준이 아니라 completed note 기준으로 세도록 바꿨다.
- forced target track을 넣어 non-target Track_* token을 마스킹하도록 바꿨다.

4. engine.py 수정

- reviewer import 실패 시 이유를 로그로 보이게 바꿨다.
- forced target track을 grammar로 내려보내도록 바꿨다.

5. reviewer.py 수정

- import 시점에 디렉토리 생성하다 죽는 부작용을 제거했다.
- 실제 write 시점에만 폴더를 만들게 바꿨다.

[13] 왜 empty middle bars가 아직 남는가에 대한 현재 가설

가설 A. inference prompt와 task pair 분포가 아직 충분히 일치하지 않는다

- 학습에서는 task pair를 넣었지만
- 추론에서는 여전히 generic continuation/track_completion 흐름 위에서
  target_track만 힌트처럼 얹는 성격이 남아 있을 수 있다

가설 B. 모델이 "한 bar라도 성공하면 나머지는 버려도 된다"는 식으로
출력을 수렴시키고 있다

- reviewer가 reject는 하지만
- regenerate를 해도 비슷한 실패 샘플을 반복한다
- 이는 모델 내부 분포가 bar-wise fill objective를 아직 못 배운다는 뜻에 가깝다

가설 C. 현재 task pair 양은 늘었지만, generic SFT와 섞였을 때
task-specific objective가 충분히 강하지 않다

- total 1330 중 task 742는 의미 있는 비중이지만
- 실제 생성에서 요구하는 형태가 더 명시적이어야 할 수 있다


[14] 지금 필요한 다음 수정 방향

우선순위 기준으로 정리하면 아래 순서가 맞다.

1. task-specific inference path를 분리

현재 generic continuation/track_completion 위에 덧씌우는 방식보다,
아예 아래와 같은 전용 태스크로 나누는 편이 맞다.

- fill_drums_from_context
- fill_strings_from_context
- fill_bass_from_chords
- fill_piano_from_chords
- fill_guitar_from_context

즉 "target_track 힌트"가 아니라
"이 악기만, 이 범위 bar를, 이 문맥을 보고 채워라"를
전용 모드로 강하게 정의해야 한다.

2. training pair와 inference prompt 포맷을 1:1로 맞추기

- 지금 학습에서 본 입력 형식과
- 실제 DAW/engine이 추론 때 넣는 입력 형식이
  조금이라도 다르면 task generalization이 약해질 수 있다

따라서 아래를 정확히 맞춰야 한다.

- task token
- start_bar / end_bar
- target_track
- context track ordering
- separator placement
- output start convention

3. bar-fill objective를 더 강하게 만들기

현재 reviewer는 사후 검사에 가깝다.
이제는 생성 중에도 아래 제약이 더 강하게 필요하다.

- 요청된 모든 bar에 completed note가 최소 1개 이상 있어야 함
- 특정 수의 empty bars가 생기면 즉시 invalid branch 처리
- first bar only collapse 패턴이 나오면 sample 자체를 폐기

4. task pair oversampling 또는 task-only SFT stage 도입

현재는 generic continuation과 task pair를 같이 학습한다.
다음 실험으로는 아래가 필요해 보인다.

- task pair에 높은 sampling weight 부여
또는
- generic SFT 후 task-only LoRA stage를 한번 더 수행

즉 "작업형 편집" 능력을 별도로 강하게 밀어줘야 한다.

우리가 원하는 "DAW 작업형 infill assistant" 구조로는
