[버그 1] SFT DataLoader KeyError — BLOCKING (신규)
================================================================

위치: midigpt/data/dataset.py, line 173

에러 메시지:
  KeyError: 'input'
  input_ids = pair["input"]

원인:
  dataset.py는 SFT 페어 JSON에서 "input" 키를 기대하지만,
  build_sft_pairs.py가 실제로 저장하는 키 이름이 다름.
  두 파일 간 스키마가 불일치하는 상황.

영향:
  SFT LoRA 학습 전체가 블로킹됨. 이 버그가 해결되지 않으면
  SFT 이후 단계로 진행 불가.

수정 제안:
  midigpt_pipeline\sft 폴더의 json 파일을 열어 실제 저장된
  키 이름을 확인한 뒤, dataset.py line 173의 pair["input"]을
  실제 키 이름으로 맞춰주면 됨. (예: "prompt", "source" 등)
  빠르면 5분 이내 수정 가능한 수준.

확인 요청:

  - EOS suppression 로직이 실제로 inference 전체 구간에 적용되고 있는지?

[버그 3] accomp 트랙 쏠림 — 지속 미해결
================================================================

현상:
  생성된 MIDI에서 드럼, 베이스, 멜로디 등 대부분의 노트가
  accomp 트랙 하나에 집중됨. 트랙 분류가 제대로 작동하지 않음.
  이전 리포트에서도 동일하게 보고됨. 개선 없음.

데이터 제작 방식 (중요):
  우리가 만든 MIDI 파일은 프로그램 넘버(program number)를 사용하지 않았고,
  채널도 악기별로 지정하지 않았음.
  트랙 분류의 유일한 기준은 아래와 같이 명시적으로 지정한 트랙 이름(track name)임.

  사용한 트랙명 목록:
    DRUM, BASS, PIANO, E.GUITAR, A.GUITAR, E.GUITAR MUTE, A.GUITAR MUTE,
    STRING, BRASS, WOODWIND, BASSDRUM, SNARE, TIMPANI, CYMBAL,
    VIOLIN1, VIOLIN2, VIOLA, CELLO, C.BASS,
    TRUMPET, F.HORN, TROMBONE, TUBA,
    FLUTE, OBOE, CLARINET, BASSOON,
    SYNTHLEAD, SYNTHPAD, SYNTHPLUCK, SYNTHBASS

확인 요청 (1):
  현재 트랙 분류 로직이 위 트랙명을 실제로 참조해서 분류하고 있는지?
  프로그램 넘버나 채널 기준으로 먼저 분류를 시도하다가 실패 후
  accomp으로 fallback하는 구조라면, 트랙명 기반 분류가
  아예 작동하지 않고 있을 가능성이 있음.

확인 요청 (2):
  위 목록에 없는 트랙명이 들어왔을 때의 처리 방식 확인 필요.
  현재는 미등록 트랙명이 들어오면 accomp으로 묶이는 것으로 보이는데,
  이 경우 accomp이라는 별도 카테고리로 생성되는 게 의도된 동작인지,
  아니면 에러 상황인지 명확히 해줬으면 함.
  의도된 동작이라면 accomp 카테고리가 너무 광범위하게 적용되고 있어
  실질적인 트랙 구분이 무의미해지는 문제가 있음.

■ 종합 품질 평가
================================================================

Pre-training 완료 후 test_generate.py로 생성 결과물을
Cubase에서 청취했으나 음악으로 들리지 않음.
이전 테스트 결과와 동일한 수준으로 개선이 없음.

SFT가 아직 완료되지 않은 영향도 있지만,
버그 2·3이 데이터 전처리 → 토크나이징 → 디코딩으로 이어지는
파이프라인 설계 단계의 문제일 가능성이 높아,
Pre-training 품질과 무관하게 출력이 개선되기 어려운 구조로 보임.
